# 입장 QR 문제점

## 문제 상황

티켓 구매자에게 가상티켓을 부여한 뒤 공연 임박 시점에 입장 QR로 전환한다. 초기안은 공연 3일 전 활성화였으나, 이후 보안과 노출 시간을 고려해 공연 1일 전 활성화를 권장안으로 본다.

동일 토큰은 특정 사용자를 알아볼 수 있으므로 보안적 입장에서 salt를 추가해 외부에서는 사용자를 알아보지 못하게 한다.

입장 QR은 모바일 캡처 방지 기능을 사용해 캡처 불가능하게 하고, 웹에서는 확인할 수 없게 구성한다. QR은 20초마다 다른 QR로 교체하고 최초 1번 사용 후에는 다시 사용할 수 없게 한다.

문제는 그래도 우회해서 캡처하거나 다른 휴대폰으로 촬영한 뒤 다른 사이트에 판매하면 어떻게 추적할 수 있느냐이다.

## 핵심 결론

완전한 캡처 방지는 불가능하다고 봐야 한다. 특히 다른 휴대폰으로 화면을 찍으면 웹이나 앱이 막을 수 없다.

그래서 핵심은 방지, 추적, 무력화를 같이 설계하는 것이다.

```text
QR 안에는 개인정보를 넣지 않음
→ 대신 랜덤한 1회성 토큰 ID만 넣음
→ 서버는 그 토큰이 어떤 티켓/구매자/세션에서 발급됐는지 알고 있음
```

외부 사람이 QR을 봐도 사용자를 알 수 없지만, 우리 서버는 해당 QR이 어느 티켓에서 나온 것인지 추적 가능해야 한다.

## QR Payload 예시

```json
{
  "jti": "random-token-id",
  "ticketId": "opaque-ticket-token",
  "iat": 1780000000,
  "exp": 1780000020,
  "sig": "server-signature"
}
```

서버 내부 DB에는 다음과 같이 저장한다.

```text
jti → ticket_id → booking_id → user_id → device_id → issued_at
```

누군가 캡처해서 판매했고 그 QR이 입장 게이트에서 찍히면 다음 정보를 추적할 수 있어야 한다.

```text
이 QR은 ticket_123에서 발급됨
ticket_123은 user_A가 구매함
발급 시간은 19:02:20
발급 기기는 iPhone 12
```

## 필요한 보안 설계

### 1. QR 20초 만료

캡처해서 판매해도 시간이 지나면 무효가 되도록 한다.

### 2. 입장 1회 사용

먼저 찍은 1명만 입장하고, 이후 같은 티켓은 차단한다.

### 3. 동적 워터마크

QR 주변에 구매자 식별용 짧은 코드, 예매번호 일부, 시간 코드를 계속 움직이게 표시한다.

예시:

```text
TG-8F3A · 19:02:20 · M-0421
```

이 값은 개인정보가 아니라 내부 추적 코드이다. 캡처 이미지가 중고 사이트에 올라오면 어떤 계정에서 나온 화면인지 추적할 수 있다.

### 4. 발급 로그 저장

QR 열람 시마다 다음 정보를 저장한다.

```text
user_id
ticket_id
device_id
ip
issued_at
jti
```

### 5. 리스크 탐지

같은 티켓 QR을 여러 기기 또는 여러 IP에서 반복 열람하면 경고한다.

공연 직전에 과도하게 QR을 열람하는 경우도 경고 또는 추가 인증 대상으로 본다.

### 6. 공식 재판매만 허용

구매자가 티켓을 팔고 싶다면 QR 캡처 판매가 아니라 플랫폼 내 공식 양도 또는 공식 재판매로 유도한다.

## Salt와 추적의 관계

salt를 추가해 외부에서는 특정 사용자를 알아볼 수 없게 해야 한다.

하지만 서버 내부에서는 반드시 추적 가능해야 한다.

```text
외부에는 사용자 식별 불가
서버 내부에서는 추적 가능
```

사용자 정보를 QR에 직접 넣지는 않되, 서버 DB에서는 어느 구매자의 티켓인지 역추적 가능해야 한다.

## 입장 QR 확인 버튼 방식

구매 직후부터 QR을 계속 보여주는 것이 아니라 다음 흐름이 더 안전하다.

```text
구매 직후: 가상 티켓만 표시
공연 1일 전: 입장 QR 확인 버튼 활성화
입장 직전: 사용자가 버튼 눌러 QR 열람
```

추천 흐름:

```text
구매 완료
→ 가상 티켓 발급
→ 공연 1일 전까지 QR 비활성화
→ 공연 1일 전 "입장 QR 확인" 버튼 활성화
→ 버튼 클릭 시 QR 화면 열림
→ QR은 20초마다 갱신
→ 입장 게이트에서 1회 사용
→ 사용 후 즉시 무효
```

버튼 문구 예시:

```text
입장 QR 확인하기
```

또는:

```text
입장 시 QR 열기
```

QR 화면 안내 문구 예시:

```text
QR은 20초마다 갱신됩니다.
캡처 또는 공유된 QR은 입장이 제한될 수 있습니다.
입장 시 직원에게 이 화면을 직접 보여주세요.
```

## 버튼 상태

| 시점 | 버튼 상태 |
| --- | --- |
| 구매 직후~공연 1일 전 | 입장 QR은 공연 1일 전부터 확인 가능합니다 |
| 공연 1일 전~공연 종료 전 | 입장 QR 확인하기 활성화 |
| 입장 완료 후 | 입장 완료 |
| 공연 종료 후 | 기간 만료 |

## 계정 양도형 암표 문제

QR 캡처보다 더 어려운 문제는 판매자가 계정 자체를 넘겨주고, 나중에 비밀번호를 다시 바꾸는 방식이다.

이 경우 단순히 QR만 추적해서는 막기 어렵다.

핵심은 계정 로그인과 입장권 사용 권한을 분리하는 것이다.

```text
계정 로그인 = 티켓 소유 증명 아님
입장 QR 열람 = 기기, 본인, 위험도 검증 후 허용
```

방어 방향은 다음과 같다.

1. 공식 양도 및 공식 재판매 기능을 제공한다.
2. 입장 QR은 신뢰된 기기 또는 정상 위험도 상태에서만 바로 열 수 있게 한다.
3. 공연 임박 시점에 비밀번호 변경, 휴대폰 번호 변경, 이메일 변경, 새 기기 로그인, 여러 지역/IP 접속이 발생하면 QR 열람을 추가 확인 대상으로 전환한다.
4. 계정 양도, 계정 판매, 비공식 티켓 거래는 약관상 금지한다.
5. 분쟁 발생 시 로그인, 기기, QR 발급, 입장 스캔 로그를 근거로 판단한다.

## 정상 사용자 불편 최소화

모든 사용자에게 강한 인증을 걸면 정상 구매자가 불편해진다.

따라서 기본은 편하게 두고, 위험 신호가 있을 때만 추가 인증을 요구한다.

```text
정상 사용자 95%는 바로 입장
수상한 5%만 추가 확인
```

| 상황 | 사용자 경험 |
| --- | --- |
| 평소 쓰던 기기 | QR 바로 열림 |
| 새 휴대폰으로 로그인 | OTP 1회 |
| 공연 1일 전 이후 비밀번호 변경 | OTP 1회 또는 추가 확인 |
| 공연 당일 새 기기 + 비밀번호 변경 | QR 보류 또는 고객센터 확인 |
| 공식 양도 완료 | 새 소유자에게 정상 권한 이전 |
| 앱 재설치 | 이메일/SMS 1회 후 정상 |

차단보다 단계적 마찰을 주는 방식이 현실적이다.

```text
risk_score < 30: 정상
30~60: OTP 1회
60~80: QR 열람 지연 또는 고객센터 확인
80 이상: QR 보류 또는 구매 제한
```

## 기기 정보 수집과 약관

기기 정보는 계정 양도와 부정예매를 탐지하는 중요한 신호가 될 수 있다.

다만 기기 정보도 개인정보 또는 개인정보에 준하는 식별정보로 볼 수 있으므로 목적, 항목, 보관기간, 동의를 명확히 해야 한다.

수집 목적 예시:

```text
부정예매 방지
계정 도용 탐지
계정 양도 및 암표 거래 탐지
입장 QR 보안
고객 분쟁 처리
```

수집 항목 예시:

```text
device_id
device_fingerprint_hash
OS 종류
브라우저/앱 버전
기기 종류
푸시 토큰
마지막 로그인 IP
마지막 로그인 시각
기기 등록 시각
```

가능하면 원본 기기정보를 그대로 저장하지 않고 해시 또는 임의 ID 형태로 저장한다.

피해야 할 정보:

```text
정확한 위치정보 상시 수집
연락처
사진/파일
광고 ID 무단 수집
IMEI 같은 고유 식별자
필요 이상 상세한 디바이스 fingerprint
```

회원가입, 로그인, 티켓 구매, 입장 QR 열람 시점의 기기 정보를 비교한다.

예시:

```text
가입 기기: iPhone
구매 기기: iPhone
입장 QR 열람: 갑자기 Android + 다른 지역 IP
→ 추가 인증
```

## 모바일 웹과 앱 전략

초기부터 반드시 앱이 필요한 것은 아니다.

하지만 입장 QR 보안, 기기 식별, 푸시 알림, 캡처 방지까지 진지하게 하려면 장기적으로 앱이 유리하다.

| 방식 | 장점 | 단점 |
| --- | --- | --- |
| 모바일 웹 | 개발 빠름, 설치 필요 없음, 접근 쉬움 | 캡처 방지, 기기 식별, 푸시, 보안 제어 약함 |
| 모바일 앱 | 기기 식별, 푸시, 캡처 방지, 보안 저장소, 생체 인증 가능 | 개발, 심사, 배포, 유지 비용 큼 |

초기 MVP는 다음 방식으로 가능하다.

```text
모바일 웹
+ 20초 회전 QR
+ 1회 사용
+ 입장 QR 1일 전 활성화
+ 열람 로그
+ 위험 시 OTP
```

고도화 단계에서는 다음 기능을 앱으로 이전한다.

```text
Ticketground 앱
+ 기기 등록
+ 푸시 인증
+ 앱 전용 QR
+ 캡처 방지/감지
+ 생체 인증
+ 보안 저장소
```

백엔드는 처음부터 웹과 앱을 모두 수용할 수 있게 설계한다.

```text
device_id
session_id
ticket_id
qr_token_id
qr_channel = WEB | APP
trusted_device
risk_score
```

## QR 1일 전 활성화와 서버 부하

입장 QR 버튼을 공연 1일 전에 활성화하는 것 자체가 서버 과부하를 만들지는 않는다.

과부하는 활성화 시점보다 사용자가 동시에 QR을 열어보는 시점에 발생한다.

가장 큰 피크는 보통 공연 당일 입장 1~2시간 전이다.

추천 구조:

```text
공연 1일 전부터 버튼 활성화
실제 QR 토큰은 버튼을 누를 때 실시간 발급
QR은 20초마다 갱신
```

QR을 1일 전에 모두 미리 생성하지 않는다.

서버 부하를 줄이는 방식:

1. 버튼 활성화와 QR 발급을 분리한다.
2. QR 발급/갱신에는 Redis 같은 단기 캐시를 사용한다.
3. QR 토큰 TTL은 20~30초로 둔다.
4. 같은 티켓에서 과도한 QR 요청이 발생하면 rate limit을 건다.
5. 입장 시간대 대형 트래픽을 기준으로 부하 테스트한다.
6. QR 자체는 사용자별 동적 데이터이므로 CDN 캐시하지 않는다.
7. 정적 UI와 이미지 리소스만 CDN을 사용한다.

활성화 시점 비교:

| 활성화 시점 | 장점 | 단점 |
| --- | --- | --- |
| 3일 전 | 사용자 여유 있음 | 캡처/거래 노출 기간 길어짐 |
| 1일 전 | 보안 좋음, 노출 짧음 | 문의 증가 가능 |
| 공연 당일 | 가장 안전 | 사용자 불안/CS 증가 |
| 입장 2~3시간 전 | 매우 안전 | 앱/서버 장애 시 리스크 큼 |

현실적인 권장안:

```text
입장 QR 버튼: 공연 1일 전 활성화
실제 QR: 사용자가 버튼 누를 때 발급
QR 갱신: 20초
위험 계정: 당일 추가 인증
서버: Redis 기반 단기 토큰 캐시
```

## 가상티켓 형식과 양도

가상티켓은 입장권이 아니라 소유권 증명서처럼 설계한다.

```text
가상티켓 = 양도 가능한 티켓 소유권
입장 QR = 공연 임박 시 생성되는 1회용 입장 권한
```

가상티켓에는 입장 QR을 넣지 않고 티켓 소유, 상태, 양도 가능 여부만 담는다.

예시:

```json
{
  "virtualTicketId": "vt_8f3a92",
  "ticketToken": "opaque_random_token",
  "eventId": "event_123",
  "seatId": "seat_A_12",
  "ownerUserId": "user_123",
  "status": "OWNED",
  "transferable": true,
  "resellable": true,
  "admissionQrAvailableAt": "2026-09-18T19:00:00+09:00",
  "createdAt": "2026-06-18T12:00:00+09:00"
}
```

사용자 화면에는 다음처럼 보여준다.

```text
TIG Live: Neon Stage
2026.09.19 19:00
잠실 올림픽 주 경기장
A구역 12열 8번
상태: 보유중
입장 QR: 공연 1일 전 활성화
[양도하기] [공식 재판매]
```

양도는 가상티켓 이미지 전달이 아니라 DB의 `owner_user_id` 변경으로 처리한다.

양도 흐름:

```text
A가 B에게 양도 요청
B가 로그인 후 수락
A의 티켓 권한 제거
B에게 가상티켓 재발급 또는 소유자 변경
양도 이력 저장
```

양도 이력:

```text
from_user_id
to_user_id
ticket_id
transferred_at
reason
ip
device_id
```

권장 상태값:

```text
OWNED
TRANSFER_PENDING
TRANSFERRED
RESALE_LISTED
RESOLD
QR_ACTIVE
USED
CANCELED
EXPIRED
LOCKED
```

양도 정책:

| 시점 | 양도 가능 여부 |
| --- | --- |
| 구매 직후~공연 1일 전 | 가능 |
| 입장 QR 활성화 후 | 제한 또는 본인확인 후 가능 |
| 공연 시작 2~3시간 전 | 양도 불가 권장 |
| 입장 완료 후 | 불가 |
| 공식 재판매 등록 중 | 직접 양도 불가 |

## 분산형 매크로 계정 대응

암표상이 계정 하나당 티켓 하나씩 구매하면 단순 계정 제한으로는 막기 어렵다.

따라서 계정 단위가 아니라 사람, 기기, 결제수단, 행동 패턴, 네트워크 단위로 묶어서 봐야 한다.

```text
계정 100개가 아니라
같은 사람 또는 같은 조직이 조작한 100개인지 탐지
```

초기 방어 조합:

```text
휴대폰 본인인증 1명당 구매 제한
결제수단 fingerprint 제한
대기열 토큰
기기 fingerprint
좌석 선점 rate limit
QR은 공연 1일 전 활성화
공식 양도/재판매 제공
```

탐지 신호:

```text
같은 본인인증 정보에서 다수 계정 생성
같은 카드/간편결제 fingerprint에서 다수 계정 구매
같은 기기에서 다수 계정 로그인
같은 IP/ASN/프록시 대역에서 대량 구매
마우스/터치 이벤트 없음
폼 입력 속도 비정상
좌석 선택 시간이 비정상적으로 빠름
여러 계정이 같은 타이밍에 같은 요청
반복 좌석 선점 후 결제 실패
```

대기열은 계정만 보지 않고 세션, 기기, IP 일부, 본인인증 정보를 같이 묶은 토큰으로 운영한다.

```text
대기열 입장
→ 브라우저/기기 토큰 발급
→ 순번 부여
→ 순번 도달 시 짧은 구매 시간 제공
→ 좌석 선점 시간 제한
→ 결제 실패/반복 선점 제한
```

완벽하게 막는 것은 어렵다.

대신 다음 방향으로 설계한다.

```text
매크로 비용 증가
계정 양도 불편
비공식 재판매 위험 증가
공식 재판매가 더 편함
```

## 최종 정리

캡처 자체를 100% 막는 것은 불가능하다.

대신 QR을 짧게 만료시키고, 1회만 사용 가능하게 만들고, QR 화면에 추적 가능한 비개인 식별 코드를 넣어야 한다.

구매 직후 보여주는 것은 소장 및 확인용 가상 티켓이고, 실제 입장권은 공연 1일 전부터 버튼으로 열람하게 하는 구조가 안전하다.

QR payload에는 개인정보를 넣지 않고 다음 정도만 넣는다.

```text
ticket_token
jti
exp
signature
```

서버 내부에서는 `jti`, `ticket_token`, `booking_id`, `user_id`, `device_id`, `issued_at`을 매핑해 추적 가능하게 유지한다.

추가로 다음 원칙을 적용한다.

```text
가상티켓은 소유권 증명서로 설계
입장 QR은 1회성 권한으로 설계
공식 양도/재판매를 제공해 계정 거래를 줄임
정상 사용자는 편하게, 위험 사용자만 추가 인증
기기 정보는 최소 수집하고 해시/임의 ID 중심으로 저장
분산형 매크로는 본인인증, 결제수단, 기기, IP, 행동 패턴을 묶어 탐지
```
