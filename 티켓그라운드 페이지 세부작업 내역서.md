# 티켓그라운드 페이지 세부작업 내역서

작성일: 2026-06-30

## 0. 문서 목적

이 문서는 `사업계획서(260624).md`를 기준으로 Ticketground 사용자 페이지에서 실제 기능들이 어떤 역할을 하는지 트리 구조로 정리한 세부 작업 내역서다. 프론트엔드는 별도 작업자가 진행한다는 전제를 유지하며, 이 문서에서는 사용자 화면, 백엔드 API, 데이터 흐름, 테스트 포인트를 기능 단위로 연결해 정리한다.

관리자 페이지 UI는 현재 세부작업 범위에서 제외한다. 단, 사용자 페이지 기능 검증에 필요한 관리자 전용 API 차단 여부나 데이터 경계는 보안 항목에서만 다룬다.

## 1. 표기 기준

```text
[화면] 사용자가 직접 접근하는 페이지 라우트
[컴포넌트] 화면을 구성하는 React 컴포넌트
[API] 백엔드 HTTP API
[백엔드] API 뒤에서 실제 도메인 처리를 담당하는 서버 파일
[데이터] 화면 또는 API가 사용하는 주요 데이터
[검증] 기능이 정상인지 확인하는 테스트 또는 QA 항목
[보강] 현재 구현 이후 추가로 세부 구현이 필요한 작업
```

```text
├─ 7. 가상 티켓, 입장 QR, 전용앱
│  ├─ 역할
│  │  ├─ 가상 티켓은 구매 사실과 소유 상태를 확인하는 용도다.
│  │  ├─ 실제 입장 QR은 공연 2~3시간 전부터 전용앱에서만 활성화한다.
│  │  ├─ QR은 20초마다 갱신되고 1회 사용 후 폐기된다.
│  │  └─ 웹은 캡처 방지를 위해 실제 입장 QR을 표시하지 않는다.
│  ├─ 사용자 행동
│  │  ├─ 구매 직후 웹/앱에서 가상 티켓을 확인한다.
│  │  ├─ 공연 1일 전 입장 QR 준비 상태와 앱 설치 안내를 확인한다.
│  │  ├─ 공연 2~3시간 전 앱 딥링크로 전용앱을 연다.
│  │  ├─ OS 생체인증 또는 기기 잠금 인증을 통과한다.
│  │  └─ 현장에서 앱의 동적 QR을 제시한다.
│  ├─ 시스템 역할
│  │  ├─ `POST /api/tickets/virtual-qr`로 가상 티켓 QR 정보를 발급한다.
│  │  ├─ `POST /api/devices/trust`로 신뢰 기기를 등록한다.
│  │  ├─ `POST /api/tickets/qr`은 APP 채널, 신뢰 기기, 앱 attestation, 공연 임박 조건을 요구한다.
│  │  ├─ `POST /api/gate/verify`는 최초 1회만 유효 처리한다.
│  │  ├─ 에뮬레이터, 원격 제어, 화면 미러링, 가상 디스플레이, 루팅/탈옥 탐지를 앱 정책에 포함한다.
│  │  └─ QR 열람 시 사용자, 티켓, 기기, IP, 시각 로그를 저장한다.
│  ├─ 관련 화면/API
│  │  ├─ [API] `POST /api/tickets/virtual-qr`
│  │  ├─ [API] `POST /api/devices/trust`
│  │  ├─ [API] `POST /api/tickets/qr`
│  │  └─ [API] `POST /api/gate/verify`
│  └─ 완료 기준
│     ├─ 웹에서는 실제 QR 버튼이 활성화되지 않는다.
│     ├─ 토큰값, nonce, signature, ownerId가 UI에 보이지 않는다.
│     ├─ QR은 20초 TTL과 1회 사용 정책을 가진다.
│     └─ 앱 설치 불가, 배터리 방전, 앱 장애는 현장 고객센터 예외 플로우로 분기한다.
│
├─ 8. 공식 재판매 및 플랫폼 내 양도
│  ├─ 역할
│  │  ├─ 사용자가 보유 티켓을 플랫폼 안에서만 다시 판매하거나 양도하게 한다.
│  │  ├─ 판매자와 구매자가 외부에서 특정 좌석을 약속해 넘기는 지정 양도를 어렵게 한다.
│  │  └─ 암표 거래보다 공식 재판매가 더 쉽고 안전한 경로가 되게 한다.
│  ├─ 판매자 행동
│  │  ├─ 보유 티켓을 선택한다.
│  │  ├─ 티켓 원가와 허용 가격 범위를 확인한다.
│  │  ├─ 판매 가격을 직접 입력한다.
│  │  └─ 공식 재판매 풀에 등록한다.
│  ├─ 구매자 행동
│  │  ├─ 공식 재판매 티켓을 확인한다.
│  │  ├─ 좌석 등급, 구역, 가격 범위, 연석 여부를 선택한다.
│  │  ├─ `재판매 티켓 구매` 버튼으로 결제와 매칭을 진행한다.
│  │  └─ 매칭 완료 후 실제 좌석번호와 총 결제액을 확인한다.
│  ├─ 시스템 역할
│  │  ├─ `POST /api/resale/list`로 보유 티켓만 등록한다.
│  │  ├─ 등록가는 원 티켓 금액의 정책 범위 안에서만 허용한다.
│  │  ├─ `POST /api/resale/purchase`로 구매자를 풀에 등록하고 결제 완료 후 자동 매칭한다.
│  │  ├─ 결제 전에는 좌석번호 직접 지정 거래를 허용하지 않는다.
│  │  ├─ 결제 실패 시 매칭 상태를 변경하지 않는다.
│  │  ├─ 판매자 자기 구매는 차단한다.
│  │  ├─ 최소 풀 크기 미만이면 조건 완화 또는 대기를 유도한다.
│  │  └─ 재판매 등록, 조건, 매칭, 수수료, 정산 예정액, 랜덤 시드를 감사 원장에 기록한다.
│  ├─ 양도 정책
│  │  ├─ Ticketground에서 구매한 티켓은 Ticketground 안에서만 양도한다.
│  │  ├─ 동반자가 따로 입장해야 할 때 공식 양도로 소유권을 이전한다.
│  │  ├─ 외부 직거래, QR 캡처 전달, 계정 통째 양도를 정식 기능으로 제공하지 않는다.
│  │  └─ 양도/재판매 결제는 초기 MVP에서 충전금 전용이 아니라 카드, 간편결제, 계좌이체 등 일반 PG 결제를 기본으로 한다.
│  ├─ 관련 화면/API
│  │  ├─ [화면] `src/app/resale/page.tsx`
│  │  ├─ [화면] `src/app/transfer/page.tsx`
│  │  ├─ [API] `POST /api/resale/list`
│  │  ├─ [API] `POST /api/resale/purchase`
│  │  ├─ [API] `POST /api/resale/join`
│  │  ├─ [API] `POST /api/resale/draw`
│  │  └─ [API] `POST /api/security/direct-transfer-attempt`
│  └─ 완료 기준
│     ├─ 홈, My 예매내역, 재판매 페이지에서 공식 재판매/양도 흐름으로 진입할 수 있다.
│     ├─ 사용자는 특정 판매자나 좌석번호를 직접 지정해 우회 거래하기 어렵다.
│     ├─ 구매 전 양도 금액, 공식 양도 수수료, 최종 결제 금액이 모두 보인다.
│     └─ 충전금 기능은 현재 기본 기능에서 제외된다.
│
├─ 9. 고객센터, 상담, 현장 예외
│  ├─ 역할
│  │  ├─ 예매, 결제, QR, 재판매, 취소/환불 문의를 처리한다.
│  │  ├─ 공연 당일 입장 QR 문제를 우선 분류한다.
│  │  └─ 1인 운영 단계에서는 내부 문의함과 카카오톡 채널을 병행한다.
│  ├─ 사용자 행동
│  │  ├─ FAQ를 확인한다.
│  │  ├─ 예매번호가 연결된 1:1 문의를 생성한다.
│  │  ├─ 상담원 답변을 스레드로 확인한다.
│  │  └─ 앱 설치 불가, 배터리 방전, 앱 장애 시 현장 고객센터 예외 입장을 요청한다.
│  ├─ 시스템 역할
│  │  ├─ 문의 생성 시 예매번호, 공연명, 티켓 상태를 함께 연결한다.
│  │  ├─ 입장 QR, 결제 실패, 공연 당일 문의를 우선순위로 분류한다.
│  │  ├─ 문의와 답변을 감사 원장 또는 상담 이력에 기록한다.
│  │  ├─ 카카오톡/문자 운영자 알림 placeholder를 생성한다.
│  │  └─ 현장 예외 입장은 본인확인, 예매정보, 결제수단 일부 확인, 운영자 승인, 1회용 처리 로그를 요구한다.
│  ├─ 관련 화면/API
│  │  ├─ [화면] `src/app/help/page.tsx`
│  │  ├─ [화면] `src/app/inquiry/page.tsx`
│  │  ├─ [API] `GET /api/support/threads?userId=...`
│  │  ├─ [API] `POST /api/support/threads`
│  │  └─ [API] `POST /api/support/messages`
│  └─ 완료 기준
│     ├─ 사용자가 24시간 실시간 상담으로 오해하지 않도록 운영시간 기준 문구를 쓴다.
│     ├─ 민감 문의는 로그인과 예매정보 확인을 거친다.
│     └─ 현장 예외가 암표 우회 통로가 되지 않도록 로그와 승인 절차를 남긴다.
│
├─ 10. 계정 위험, 매크로, 암표 우회 대응
│  ├─ 역할
│  │  ├─ 매크로 예매, 다계정 조직 구매, 계정 통째 양도, QR 캡처 판매를 줄인다.
│  │  ├─ 정상 사용자는 추가 인증으로 통과시키고 고위험 계정만 단계적으로 제한한다.
│  │  └─ 자동 탐지와 운영자 검토, 고객센터 소명 절차를 함께 둔다.
│  ├─ 탐지 신호
│  │  ├─ 같은 본인인증 정보의 다계정 생성
│  │  ├─ 같은 결제수단 fingerprint의 반복 구매
│  │  ├─ 같은 기기에서 여러 계정 로그인
│  │  ├─ 프록시, VPN, 데이터센터 IP 대역 반복
│  │  ├─ 비정상적으로 빠른 좌석 선택과 이벤트 부족
│  │  ├─ 결제 실패 반복
│  │  └─ 공연 1일 전 이후 비밀번호 변경, 연락처 변경, 새 기기 로그인, QR 열람 기기 변경 동시 발생
│  ├─ 시스템 조치
│  │  ├─ 1단계: 대기열 재검증, CAPTCHA, OTP
│  │  ├─ 2단계: 구매 수량 제한, 좌석 선점 시간 단축, 구매 직후 OTP 또는 권한 보류
│  │  ├─ 3단계: 예매 보류, 입장 QR 활성화 보류, 고객센터 확인
│  │  ├─ 4단계: 티켓 취소, 계정 주의/차단, 재판매 제한, 증빙 보관
│  │  └─ 5단계: 반복 조직 행위는 영구 차단과 신고 검토
│  ├─ 관련 데이터
│  │  ├─ 요청 로그
│  │  ├─ 기기 로그
│  │  ├─ 네트워크 로그
│  │  ├─ 결제 로그
│  │  ├─ 행동 로그
│  │  └─ 티켓/QR/재판매 로그
│  └─ 완료 기준
│     ├─ 자동 탐지만으로 최종 제재하지 않는다.
│     ├─ 고위험 계정은 QR 활성화를 임시 보류할 수 있다.
│     ├─ 정상 사용자는 OTP 또는 고객센터 확인으로 복구할 수 있다.
│     └─ 모든 제재 근거는 분쟁 대응 가능한 로그로 남는다.
│
├─ 11. 결제, 정산, 수익 모델
│  ├─ 역할
│  │  ├─ 예매 수수료, 공식 양도 수수료, 공식 재판매 수수료, 프로모션, 운영 도구 수익을 구분한다.
│  │  ├─ 취소 수수료나 충전금 환급 수수료를 핵심 수익원으로 삼지 않는다.
│  │  └─ 초기 MVP는 일반 PG 결제와 정산 계좌 지급 구조로 단순하게 운영한다.
│  ├─ 사용자에게 보여야 하는 정보
│  │  ├─ 티켓 금액
│  │  ├─ 예매 수수료
│  │  ├─ 공식 양도 수수료
│  │  ├─ 최종 결제 금액
│  │  ├─ 환불 예상 금액
│  │  └─ 취소/환불 기준
│  ├─ 시스템 역할
│  │  ├─ PG 승인, 취소, 환불, 정산 상태를 관리한다.
│  │  ├─ 결제 실패 시 예매 또는 재판매 매칭을 완료 처리하지 않는다.
│  │  ├─ 양도자는 판매 확정 후 정산 계좌로 지급한다.
│  │  ├─ 플랫폼은 양도 완료 시점에 투명한 공식 양도 수수료를 받는다.
│  │  └─ 충전금은 추후 빠른 매칭, 빠른 정산, 팬 월렛 옵션으로만 검토한다.
│  └─ 완료 기준
│     ├─ 사용자가 결제 전 최종 금액을 이해할 수 있다.
│     ├─ 결제 실패가 좌석/티켓/재판매 상태를 오염시키지 않는다.
│     └─ 충전금 도입 전까지 환급/잔액/현금 전환 UI가 노출되지 않는다.
│
├─ 12. 데이터, API, 감사 원장
│  ├─ 역할
│  │  ├─ 로컬 JSON DB 기준 users, venues, events, tickets, resalePools, watchlist, supportThreads, admissionCredentials 등을 관리한다.
│  │  ├─ 구매, 재판매, QR, 제재, 문의, 알림 이벤트를 append-only 해시 체인 원장에 기록한다.
│  │  └─ 사용자 공개 API와 관리자 API의 데이터 경계를 분리한다.
│  ├─ 사용자 공개 API 원칙
│  │  ├─ 티켓 ownerId를 노출하지 않는다.
│  │  ├─ admission credential, QR nonce, signature, 내부 token을 노출하지 않는다.
│  │  ├─ 잔액, 신뢰 점수, 제재 이력은 공개 상태 응답에서 제외한다.
│  │  └─ 관리자 API는 사용자 포트에서 404로 차단한다.
│  ├─ 감사 원장 기록 대상
│  │  ├─ 최초 데이터 생성
│  │  ├─ 티켓 구매
│  │  ├─ 공식 재판매 등록
│  │  ├─ 재판매 매칭
│  │  ├─ 지정 양도 차단
│  │  ├─ QR 토큰 발급/검증
│  │  ├─ 관심 공연/알림
│  │  └─ 고객 문의/답변
│  └─ 완료 기준
│     ├─ 사용자 화면에서 민감 데이터가 보이지 않는다.
│     ├─ `GET /api/ledger/verify`로 원장 무결성 확인이 가능하다.
│     └─ 사용자 기능에 필요한 공개 DTO가 프론트엔드 타입과 맞는다.
│
└─ 13. 외부 연동 및 출시 전 체크
   ├─ PG 연동
   │  ├─ 결제 승인
   │  ├─ 결제 취소
   │  ├─ 부분 환불
   │  ├─ 주최사 정산
   │  └─ 재판매 판매자 정산
   ├─ 카카오톡/문자/앱 푸시
   │  ├─ 관심 공연 예매 오픈 D-3 알림

   │  ├─ 입장 QR 준비 안내

   │  └─ 고객 문의 답변 알림
   ├─ 사용자 전용앱
   │  ├─ 로그인/기기 확인
   │  ├─ 생체인증 또는 기기 잠금 인증
   │  ├─ 앱 attestation
   │  ├─ 실제 입장 QR 표시
   │  └─ 가상화면 실행 차단
   ├─ 현장 게이트 앱

   │  ├─ 실시간 검증
   │  ├─ 사용 처리

   └─ 출시 전 체크
      ├─ 약관, 개인정보처리방침, 결제/환불 정책
      ├─ 관리자 접근 제어와 로그 보관

      ├─ 부하 테스트와 대기열 검증
      └─ 현장 스태프 교육과 장애 대응 절차
```

## 3. 사용자 페이지 전체 기능 트리

```text
Ticketground 사용자 페이지

├─ 2. 공연 탐색 및 검색
│  ├─ 2.1 메인 홈
│  │  ├─ 역할
│  │  │  ├─ Ticketground의 첫 진입 화면이다.
│  │  │  ├─ 주요 공연, 오픈 예정, 랭킹, 추천 공연, 프로모션으로 사용자를 예매 상세로 유도한다.

│  │  │  ├─ 사용자가 구매하지 못한 티켓은 공식 재판매 풀에서 찾고, 보유 티켓은 플랫폼 안에서 안전하게 넘길 수 있다는 메시지를 제공한다.
│  │  │  └─ 플랫폼이 작아 보이지 않도록 넓은 화면 구성과 충분한 공연 수를 보여준다.

│  │     ├─ 홈에 `티켓 재판매` 또는 `공식 재판매/양도` 섹션을 추가해 `/resale`, `/transfer`로 이동할 수 있게 한다.
```

## 4. 우선순위별 세부 작업 트리

```text
P0. 사용자 페이지와 백엔드 연결 안정화
├─ 메인 홈에 공식 재판매/플랫폼 내 양도 섹션 추가
├─ 홈의 재판매/양도 CTA를 `/resale`, `/transfer`로 연결
├─ 예매 완료 버튼을 `POST /api/tickets/buy`와 연결
├─ My 예매내역을 `GET /api/users/:userId/tickets`와 연결
├─ 가상 티켓 발급을 `POST /api/tickets/virtual-qr`와 연결
├─ 웹 실제 입장 QR 버튼은 항상 비활성 유지
├─ 공식 재판매 등록/구매는 현재 연결 상태 유지 및 오류 메시지 보강
└─ 충전금 결제수단 노출 및 호출 제거 상태 유지

P1. 좌석 선택 기능 현실화
├─ `GET /api/events/:eventId/seat-map` 응답을 좌석 선택 컴포넌트에 연결
├─ 공연장별 좌석 배치 분기
├─ 판매중/매진/보류/선택중 상태 표시
├─ 좌석 선점 API 설계
├─ 결제 실패 시 좌석 선점 해제

P2. 입장 QR 보안 정책 구현
├─ 웹: 가상 티켓만 표시

├─ 앱: 신뢰 기기 등록
├─ 앱: OS 생체인증 또는 기기 잠금 인증 결과 확인
├─ 앱: 앱 attestation 서명 확인
├─ 앱: 가상화면/원격제어/미러링/루팅/탈옥 탐지
├─ QR: 공연 2~3시간 전부터 활성화

└─ 게이트: 1회 사용 후 재사용 차단

P3. 공식 재판매 완성도 보강
├─ 홈 진입점
│  ├─ 메인 홈에서 공식 재판매 기능을 독립 섹션으로 노출
│  ├─ 구매자는 공식 재판매 풀에서 티켓을 찾을 수 있음을 안내
│  ├─ 판매자는 보유 티켓을 정책 가격 안에서 등록할 수 있음을 안내
│  └─ 동반자 또는 다른 사용자에게 넘기는 경우 플랫폼 내 양도만 허용한다는 정책 안내
├─ 등록 화면
│  ├─ 티켓 원가 표시
│  ├─ 허용 등록가 범위 표시
│  ├─ 사용자가 직접 금액 입력
│  ├─ 수수료 및 정산 예정액 표시
│  └─ 정책 위반 등록 차단
├─ 구매 화면
│  ├─ 등급/구역/가격/연석 조건 선택
│  ├─ `재판매 티켓 구매` 단일 버튼 유지
│  ├─ 결제 성공 후 자동 매칭
│  ├─ 매칭 완료 후 좌석번호 공개

└─ FAQ
```

## 7. 작업 시 주의사항

```text
프론트엔드
├─ 현재 완성본 UI 형태를 우선 유지한다.
├─ 백엔드 기능 테스트에 필요한 연결만 최소 변경한다.
├─ 관리자 페이지 UI는 별도 지시 전까지 수정하지 않는다.
└─ 모바일 화면에서 실제 입장 QR 버튼이 활성화되지 않게 한다.

백엔드
├─ 사용자 포트는 4173을 기본으로 유지한다.
├─ 관리자 포트는 50084를 별도 유지한다.
├─ 사용자 API에서 관리자 데이터와 민감값을 노출하지 않는다.
├─ 충전금 결제수단은 현재 지원하지 않는다.
└─ QR, 재판매, 문의, 관심공연 이벤트는 감사 원장에 기록한다.

테스트
├─ 변경 후 `npm run check`를 기준 검증으로 실행한다.
├─ UI가 변경되면 모바일/데스크톱 시각 QA를 수행한다.
├─ API가 변경되면 backend 테스트와 frontend integration 테스트를 함께 수정한다.
└─ `DESIGN 2.md`와 무관한 파일은 커밋 대상에서 제외한다.
```
