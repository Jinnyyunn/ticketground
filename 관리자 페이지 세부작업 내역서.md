# 관리자 페이지 세부작업 내역서

작성일: 2026-06-30

## 0. 문서 목적

이 문서는 Ticketground 관리자 페이지에서 필요한 기능을 사용자 페이지 흐름에 맞춰 정리한 세부작업 내역서다.

관리자 페이지는 단순히 공연 정보를 입력하는 화면이 아니라, 사용자 페이지의 공연 상세, 예매 날짜, 좌석 선택, 결제, My 예매내역, 공식 재판매, 입장 QR 정책까지 연결하는 운영 콘솔이다.

관리자 페이지는 사용자 페이지에 노출하지 않는다. 사용자 페이지 포트에서는 `/admin.html` 같은 관리자 진입점이 보이면 안 되며, 관리자 페이지는 별도 관리자 포트 또는 별도 관리자 도메인에서 운영한다.

## 1. 운영 원칙

```text
관리자 페이지 운영 원칙
├─ 관리자와 사용자 화면은 분리한다.
├─ 관리자 기능은 사용자 페이지 메뉴, 버튼, 검색 결과에 노출하지 않는다.
├─ 티켓 판매 등록 정보는 사용자 페이지 상품 상세와 좌석 선택에 직접 반영된다.
├─ 개최 장소 선택은 좌석도 API와 연결되어야 한다.
├─ 판매 정보 수정은 감사 로그에 남긴다.
├─ 가격, 좌석, 구매 제한, QR 정책은 게시 전 검증한다.
└─ 계정 위험 관리는 내부 점수, 상태, 사유, 증빙을 함께 보고 판단한다.
```

## 4. 개최 장소 및 좌석도 API 연동

관리자 페이지에서 개최 장소를 선택하면 해당 장소의 좌석도 API가 연결되어야 한다. 사용자가 예매 화면에서 보는 좌석도는 관리자 페이지에서 선택한 `venueId`와 동일한 값을 기준으로 표시한다.

```text
장소 선택 및 좌석도 연결 흐름
├─ 관리자 페이지 접속
├─ 티켓 판매 등록 화면 열기
├─ 개최 장소 선택
├─ GET /api/admin/venues
├─ venueId 선택
├─ GET /api/seat-map?venueId={venueId}
├─ 관리자 화면에서 좌석도 미리보기
├─ 좌석 등급, 가격, 판매 제외석, 보류석 설정
├─ POST /api/admin/events/sale
├─ event.venueId 저장
├─ 사용자 페이지 상품 상세 진입
├─ 사용자 예매 날짜/회차 선택
├─ GET /api/events/{eventId}/seat-map 또는 GET /api/seat-map?eventId={eventId}
└─ 사용자 좌석 선택 화면에 같은 공연장 좌석도 표시
```

관련 API는 다음 기준으로 정리한다.

| API | 역할 |
| --- | --- |
| `GET /api/admin/venues` | 관리자 공연장 목록 조회 |
| `GET /api/seat-map?venueId=...` | 장소 기준 좌석도 미리보기 |
| `GET /api/seat-map?eventId=...` | 행사 기준 좌석도 조회 |
| `GET /api/events/:eventId/seat-map` | 사용자 예매 화면 좌석도 조회 |
| `POST /api/admin/events/sale` | 판매 정보, 날짜, 장소, 가격 저장 |

완료 기준은 다음과 같다.

```text
좌석도 연동 완료 기준
├─ 블루스퀘어 선택 시 블루스퀘어 좌석도가 관리자 미리보기에 표시된다.
├─ 잠실 올림픽 주 경기장 선택 시 잠실 주경기장 좌석도가 관리자 미리보기에 표시된다.
├─ 잠실 실내체육관 선택 시 실내체육관 좌석도가 관리자 미리보기에 표시된다.
├─ 선택한 venueId가 event에 저장된다.
├─ 사용자 페이지 좌석 선택 화면이 event.venueId 기준 좌석도를 불러온다.
├─ 관리자에서 설정한 좌석 가격이 사용자 결제 금액에 반영된다.
└─ 장소 변경 시 기존 좌석 가격/보류석 설정을 다시 검수하게 한다.
```

```text
├─ 결제 실패 건수
├─ 공식 재판매 등록 건수

```

## 6. 판매 재고 및 좌석 상태 관리

판매 재고 관리는 티켓 판매 등록 이후 운영 중에 필요한 기능이다.

```text
판매 재고 및 좌석 상태 관리
├─ 공연별 전체 좌석 수
├─ 판매 가능 좌석 수
├─ 판매 완료 좌석 수
├─ 결제 대기 좌석 수
├─ 보류 좌석 수
├─ 관계자석/초대석 수
├─ 등급별 판매율
├─ 구역별 판매율
├─ 좌석 상태 수동 변경
├─ 좌석 선점 만료 처리
└─ 좌석 상태 변경 감사 로그
```

주의할 점은 판매된 티켓 가격을 임의로 바꾸면 안 된다는 것이다. 가격 변경은 미판매 좌석 또는 향후 회차에만 적용하는 것을 기본 정책으로 한다.

## 7. 공식 재판매 및 양도 관리

관리자는 공식 재판매가 정상적으로 운영되는지 확인해야 한다.

```text
공식 재판매 및 양도 관리
├─ 공연별 재판매 허용 여부
├─ 재판매 등록 티켓 목록
├─ 재판매 가격 범위 검증
├─ 재판매 구매 조건
├─ 자동 매칭 결과
├─ 판매자 자기 구매 차단 로그
├─ 특정 좌석 지정 거래 의심 로그
├─ 양도 가능/불가 상태
├─ 양도 횟수 제한
├─ 재판매 마감 시간
└─ 분쟁 발생 시 감사 원장 조회
```

재판매와 양도는 사용자끼리 외부에서 직접 연락해 거래하는 기능이 아니다. 관리자 페이지도 이 원칙을 유지해야 하며, 외부 계좌 거래나 개인 연락처 교환을 유도하는 운영 기능은 제공하지 않는다.

## 8. 계정 상태 및 암표 대응

계정당 위험 점수는 0~100점 기준의 내부 `riskScore`로 계산한다. 다만 점수 숫자만으로 자동 제재하지 않고, 관리자 화면에서는 점수 구간, 계정 상태, 위험 사유, 증빙 로그를 함께 보여준다.

위험 계정의 기준은 다음과 같이 둔다.

```text
위험 계정 기준
├─ 0~39점: 정상 또는 관찰 계정
├─ 40~59점: 주의 계정
├─ 60점 이상: 위험 계정
├─ 80점 이상: 고위험 계정
└─ 90점 이상 또는 명확한 증빙 존재: 차단 검토 계정
```

핵심 기준은 명확하다. **60점 이상부터 위험 계정으로 분류**하고, **80점 이상부터 고위험 계정으로 분류**한다. 단, 60점 이상이라고 해서 바로 영구 차단하지 않는다. 점수는 운영자 검토를 위한 우선순위이며, 최종 조치는 증빙 로그와 관리자 확인을 거쳐야 한다.

```text
계정 위험 관리 원칙
├─ 내부 시스템은 riskScore를 계산할 수 있다.
├─ 관리자 목록 화면은 계정 상태와 위험 사유를 우선 보여준다.
├─ 관리자 상세 화면은 riskScore, 점수 구간, 산정 사유를 함께 보여준다.
├─ 최종 제재는 자동 점수만으로 확정하지 않는다.
├─ 정상 사용자 피해를 줄이기 위해 소명 절차를 둔다.
└─ 공연 당일 고위험 계정은 입장 QR 활성화를 임시 보류할 수 있다.
```

관리자 화면에서 보여줄 점수 구간과 기본 상태는 다음과 같다.

| 점수 | 상태 | 위험도 | 판단 기준 | 기본 조치 |
| --- | --- | --- | --- | --- |
| 0~19 | 정상 | 낮음 | 일반적인 가입, 예매, 결제 패턴 | 별도 조치 없음 |
| 20~39 | 관찰 | 낮음~중간 | 일부 반복 행동은 있으나 악용 정황 부족 | 로그 관찰, 자동 제재 없음 |
| 40~59 | 주의 | 중간 | 다계정, 빠른 예매 시도, 결제 실패 반복 등 일부 신호 누적 | 인기 공연 추가 인증, 구매 제한 후보 |
| 60~79 | 제한 검토 | 위험 | 매크로 의심, 동일 기기/결제수단 반복, 재판매 악용 신호 누적 | 구매 제한, QR 활성화 보류 검토, 고객센터 확인 |
| 80~89 | 고위험 | 높음 | 여러 위험 신호가 동시에 확인됨 | 예매 보류, QR 활성화 보류, 운영자 승인 필요 |
| 90~100 | 차단 검토 | 매우 높음 | 명확한 우회, 매크로, 계정 양도, QR 부정 사용 증빙 | 계정 차단 검토, 티켓 취소 검토, 증빙 보관 |

관리자 상태 값은 점수 구간과 연결하되, 수동 조정이 가능해야 한다.

| 상태 | 의미 | 가능한 조치 |
| --- | --- | --- |
| 정상 | 0~19점 또는 정상 이용 확인 | 별도 조치 없음 |
| 관찰 | 20~39점 또는 가벼운 반복 신호 | 모니터링, 자동 제재 없음 |
| 주의 | 40~59점 또는 의심 신호 누적 | 추가 인증, 구매 제한 후보 |
| 제한 | 60~79점 또는 위험 신호 누적 | 구매 제한, QR 보류, 고객센터 확인 |
| 고위험 | 80~89점 또는 복합 위험 신호 | 예매 보류, QR 활성화 보류, 운영자 승인 |
| 차단 | 90점 이상 또는 명확한 증빙 | 계정 차단, 티켓 취소 검토, 증빙 보관 |

점수는 다음 항목을 합산하되, 총점은 100점을 넘지 않게 제한한다.

| 점수 항목 | 최대 점수 | 상세 기준 |
| --- | ---: | --- |
| 계정/인증 신호 | 10점 | 신규 계정, 이메일/휴대폰 인증 미완료, 짧은 기간 내 정보 변경 반복 |
| 접속/대기열 신호 | 20점 | 비정상 요청 속도, 대기열 우회 시도, 자동화 도구 패턴, API 직접 호출 반복 |
| 구매/결제 신호 | 20점 | 인기 공연 반복 시도, 결제 실패 반복, 동일 결제수단 다계정 사용, 구매 직후 취소 반복 |
| 기기/네트워크 신호 | 15점 | 동일 기기 다계정, 데이터센터/VPN/프록시 의심, 원격 제어/가상화면 환경 |
| 재판매/양도 신호 | 15점 | 구매 직후 반복 재판매, 외부 지정 양도 시도, 가격 정책 위반, 판매자 자기 구매 시도 |
| QR/입장 신호 | 10점 | 여러 기기에서 QR 열람, 짧은 시간 내 반복 열람, 앱 보안 환경 실패, QR 부정 사용 의심 |
| 고객센터/분쟁 신호 | 10점 | 반복 예외 요청, 사회공학 의심, 본인확인 회피, 분쟁 반복 |

점수 산정 예시는 다음과 같다.

```text
riskScore 산정 예시
├─ 가입 48시간 이내 인기 공연 구매 시도 반복: +5
├─ 이메일 또는 휴대폰 인증 미완료 상태로 구매 시도: +5
├─ 초당 비정상 요청 또는 자동화 패턴 감지: +15
├─ 대기열 우회 파라미터 또는 비정상 세션 감지: +20
├─ 동일 기기에서 3개 이상 계정이 같은 공연 구매 시도: +10
├─ 동일 기기에서 5개 이상 계정이 같은 공연 구매 시도: +15
├─ 동일 결제수단이 여러 계정에서 반복 사용됨: +10~15
├─ 결제 실패 후 즉시 재시도 패턴이 반복됨: +5~10
├─ 구매 직후 대부분의 티켓을 반복 재판매 등록: +10
├─ 외부 지정 양도 또는 개인 연락처 교환 시도: +15
├─ 판매자 자기 구매 시도: +15
├─ QR을 여러 기기 또는 여러 IP에서 반복 열람: +10
├─ 원격 제어, 미러링, 가상화면, 루팅/탈옥 의심: +10~15
└─ 고객센터에서 본인확인을 반복 회피하거나 예외 처리를 반복 요구: +5~10
```

반대로 정상 이용 증빙이 있으면 점수를 낮출 수 있다.

```text
riskScore 완화 기준
├─ 휴대폰 본인확인 또는 OTP 통과: -5~-10
├─ 고객센터 본인확인 완료: -5~-15
├─ 이전 공연 정상 입장 이력 다수: -5
├─ 30일 이상 추가 위험 신호 없음: -5~-10
├─ 결제 실패가 PG 장애로 확인됨: 해당 항목 제거
└─ 가족/동반자 구매 등 정상 사유가 증빙됨: 관리자 검토 후 감점
```

즉시 고위험으로 올릴 수 있는 신호도 따로 둔다. 아래 항목은 점수가 낮아도 운영자 검토 큐에 바로 올린다.

```text
즉시 고위험 검토 신호
├─ 매크로 도구 사용 증빙
├─ 대기열 우회 또는 서명 조작 증빙
├─ 계정 통째 양도 정황 증빙
├─ QR 캡처 판매 게시글 또는 외부 판매 증빙
├─ 결제 도용 또는 차지백 악용 정황
├─ 동일 조직으로 보이는 다계정 대량 구매 정황
└─ 고객센터를 속여 QR 또는 입장 권한을 풀려는 정황
```

위험 사유는 태그 형태로 관리한다.

```text
위험 사유 태그
├─ 다계정 의심
├─ 매크로 패턴
├─ 대기열 우회 의심
├─ 동일 기기 반복 구매
├─ 동일 결제수단 반복 구매
├─ 반복 취소
├─ 재판매 정책 위반
├─ 계정 통째 양도 의심
├─ QR 이상 열람
├─ 원격 제어/가상화면 의심
└─ 고객센터 사회공학 의심
```

관리자 조치에는 반드시 사유와 증빙이 연결되어야 한다.

```text
관리자 조치 흐름
├─ 위험 신호 확인
├─ riskScore와 점수 구간 확인
├─ 증빙 로그 확인
├─ 계정 상태 선택: 정상 / 관찰 / 주의 / 제한 / 고위험 / 차단
├─ 조치 사유 선택
├─ 관리자 메모 입력
├─ "수정하시겠습니까?" 확인 팝업
├─ POST /api/admin/users/status
├─ 감사 로그 기록
└─ 필요 시 사용자 소명 또는 고객센터 플로우 연결
```

## 9. 고객 문의 관리

고객 문의는 예매, 결제, QR, 재판매, 환불 문의를 운영자가 처리하는 화면이다.

```text
고객 문의 관리
├─ 문의 목록
├─ 문의 상태: 접수 / 처리중 / 답변완료 / 보류
├─ 긴급도: 입장 QR / 결제 실패 / 공연 당일 / 일반
├─ 예매번호 연결
├─ 티켓 상태 연결
├─ 결제 상태 연결
├─ 사용자 문의 내용
├─ 관리자 답변
├─ 카카오톡 채널 연결 여부
├─ 답변 완료 알림 발송
└─ 상담 이력 감사 로그
```

1인 운영 단계에서는 모든 문의를 실시간 응대로 약속하면 안 된다. 운영 가능한 문구는 "운영시간 내 빠른 답변", "공연 당일 입장 문의 우선 응대", "답변 완료 시 알림 발송" 기준으로 잡는다.

## 10. 입장 QR 및 현장 운영 관리

관리자는 공연별 입장 QR 정책과 현장 예외 처리를 볼 수 있어야 한다.

```text
입장 QR 및 현장 운영 관리
├─ 공연별 QR 준비 상태
├─ 공연 1일 전 준비 안내 발송 여부
├─ 공연 2~3시간 전 실제 QR 활성화 여부
├─ 실제 QR 앱 전용 정책 여부
├─ 웹 실제 QR 차단 상태
├─ QR 발급 로그
├─ QR 검증 로그
├─ 1회 사용 후 폐기 여부
├─ 앱 설치 불가 예외 처리
├─ 배터리 방전 예외 처리
├─ 앱 장애 예외 처리
└─ 현장 고객센터 승인 로그
```

실제 입장 QR은 사용자 웹에 노출하지 않는 것을 기본 정책으로 유지한다. 관리자 페이지에서도 실제 QR 토큰 원문을 불필요하게 노출하지 않는다.

## 11. 결제, 환불, 정산 관리

결제 관리는 PG 연동 이후 운영 리스크가 큰 영역이다.

```text
결제, 환불, 정산 관리
├─ 예매 결제 내역
├─ 결제 실패 내역
├─ 환불 요청 내역
├─ 취소 수수료 계산
├─ 공식 재판매 결제 내역
├─ 공식 재판매 수수료
├─ 판매자 정산 예정액
├─ 정산 계좌 확인
├─ PG 승인번호
├─ PG 취소번호
├─ 세금계산서 또는 정산 증빙
└─ 결제/환불 감사 로그
```

충전금은 현재 기본 결제 기능에서 제외한다. 향후 빠른 매칭, 빠른 정산, 팬 월렛 기능으로 검토할 수 있지만, 초기 관리자 페이지에는 충전금 충전/환급/잔액 관리 기능을 기본 노출하지 않는다.

## 14. 완료 기준

```text
관리자 페이지 완료 기준
├─ 관리자 페이지가 사용자 페이지에 노출되지 않는다.
├─ 티켓 판매 등록 시 공연명, 카테고리, 장소, 날짜, 시간, 가격, 구매 제한을 입력할 수 있다.

├─ 계정 위험 점수는 0~100점 기준으로 산정된다.
├─ 60점 이상은 위험 계정, 80점 이상은 고위험 계정으로 분류된다.
├─ 계정 관리는 점수 숫자만이 아니라 상태/사유/증빙 로그 중심으로 운영된다.
├─ 고객 문의와 QR 예외 처리를 관리자 화면에서 추적할 수 있다.
└─ 모든 관리자 수정은 감사 로그로 남는다.
```
