# Ticketground 구성안

## 방향

사업계획서의 블록체인/NFT 표현은 MVP 단계에서 다음 구조로 치환했습니다.

| 계획서 표현 | 현실형 MVP 구성 |
| --- | --- |
| NFT 티켓 | 중앙 DB 티켓 레코드 |
| 스마트 계약 | 백엔드 정책 엔진 |
| 온체인 거래 기록 | append-only 해시 체인 감사 원장 |
| 개인 지갑 | 실명 계정 기반 플랫폼 보관 지갑 |
| 전송 승인 서명 | 서버 HMAC 서명 및 API 승인 |
| 장외 전송 차단 | 지정 양도 API 부재 + 차단/제재 로그 |

이 방식은 완전한 탈중앙화는 아니지만, 서비스 초기 예산에서 필요한 것은 탈중앙화보다 “플랫폼 밖에서 프리미엄을 붙여 거래할 수 없게 만드는 운영 규칙”입니다.

## 프론트엔드

- 1차 판매 티켓 목록
- 구매자 계정 선택 및 티켓 구매
- 보유 티켓의 공식 재판매 풀 등록
- 재판매 풀 대기 참여 및 랜덤 매칭 실행
- 지정 양도 시도 차단 시뮬레이션
- 동적 QR 발급 및 게이트 검증
- 해시 체인 감사 원장 확인

초기 화면은 마케팅 랜딩이 아니라 운영 대시보드 형태로 구성했습니다. 실제 사업 검증에서는 기획사/운영자/투자자가 핵심 정책을 바로 확인할 수 있어야 하기 때문입니다.

## 백엔드

### 핵심 모듈

- `tickets`: 티켓 발행, 소유자, 상태, 정가, 재판매 상한/하한, 양도 횟수 관리
- `users`: 신뢰 점수, 계정 상태, 제재 내역 관리
- `resalePools`: 구역별 공용 재판매 풀, 대기자, 매칭 결과 관리
- `ledger`: 모든 중요 이벤트를 이전 해시와 연결해 기록
- `qr`: 20초 만료 HMAC 동적 입장 토큰
- `security`: 지정 양도 시도 차단 및 신뢰 점수 차감

### 정책

- 1:1 지정 양도는 정상 거래 경로로 제공하지 않습니다.
- 재판매는 반드시 공용 풀에 등록됩니다.
- 구매자는 판매자를 지정할 수 없고, 판매자도 구매자를 지정할 수 없습니다.
- 재판매 가격은 티켓별 `minPrice`와 `maxPrice` 사이만 허용됩니다.
- 티켓 양도 횟수는 `maxTransferCount`로 제한됩니다.
- 정책 위반 시도는 원장에 남고 신뢰 점수가 차감됩니다.

## 실제 서비스화 시 보강

- DB: PostgreSQL, Prisma/Drizzle, 행 단위 감사 테이블
- 원장 보존: S3 Object Lock, WORM 스토리지, 외부 타임스탬프 공증
- 결제: PG, 에스크로, 예치금 분리 보관, 정산/환불 회계
- 본인인증: 휴대폰/KYC, 계정 중복 탐지
- 보안: rate limit, device fingerprint, IP 위험 점수, 봇 탐지
- 운영: 신고/보증금 동결/환수 어드민, 현장 게이트 앱
- 규제: 전자금융거래법, 개인정보보호법, 약관/환불 정책 검토
