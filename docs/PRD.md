# Ticketground — Product Requirements Document (PRD)

> 본 문서는 Codex(또는 임의의 코딩 에이전트)에게 **Ticketground 프런트엔드 프로토타입**을 재현·확장하도록 지시하기 위한 제품 요구사항 문서다.
> 함께 제공되는 문서: `SPEC.md`(기술 명세·정확한 수치), `PROMPT.md`(실행 지시문), `Design.md`(디자인 토큰 원본), `사업계획서(260624).md`(원본 비즈니스 PRD).

---

## 0. 한 줄 요약

콘서트·뮤지컬·연극·클래식 티켓을 예매하고, **공식 재판매·양도·동적 입장 QR로 암표를 통제**하는 "클린 티켓" 플랫폼의, 데스크톱 웹 하이파이 인터랙티브 프로토타입(정적 HTML/CSS/JS, 빌드 스텝 없음).

---

## 1. 배경과 목표

### 1.1 배경
기존 티켓 예매 서비스의 익숙한 흐름은 유지하되, 플랫폼 밖 고가 양도·계정 거래·QR 캡처 판매·매크로 예매를 줄이는 것이 핵심 경쟁력이다. 이 프로토타입은 실제 백엔드 없이도 그 차별화 기능(공식 재판매 풀, 조건부 랜덤 매칭, 가상 티켓과 동적 QR 분리, 감사 원장, 운영 콘솔)을 **화면으로 증명**하는 것을 목표로 한다.

### 1.2 목표 (이 산출물이 달성해야 할 것)
1. 대형 티켓 서비스처럼 보이는 **완성형 예매 UX** — 탐색→상세→대기열→예매→결제→완료→마이페이지.
2. 암표 통제 차별화 기능의 **시각적 증명** — 공식 재판매, 공식 양도, 가상/동적 QR 단계, 운영 콘솔, 감사 원장.
3. `Design.md`의 디자인 토큰을 기반으로 한 **일관된 미니멀·기능적 비주얼 시스템**.
4. 모든 화면이 상호 링크된 **클릭 가능한 단일 사이트** (별도 서버 불필요, 파일 더블클릭으로 열림).

### 1.3 비목표 (Out of Scope)
- 실제 PG 결제 연동, 실제 로그인/세션, 실제 DB.
- 전용앱(네이티브) 생체인증, 현장 게이트 스캐너 앱.
- 모바일 레이아웃 (데스크톱 1280px 고정 — 단, 명세에 모바일 대응을 별도 표기한 좌석 선택 제외).
- 서버 사이드 렌더링·프레임워크 빌드 파이프라인. (Tweaks 패널 한정으로만 React+Babel 인라인 사용)

## 3. 브랜드 & 톤

- **브랜드명**: Ticketground (로고 워드마크 + 우측 작은 빨강 점 `●`). 한국어 UI.
- **성격**: 깔끔하고 기능적, 미니멀. AI slop 금지(과한 그라데이션 배경, 이모지 남발, 둥근 모서리+좌측 보더 액센트 카드 등 금지).
- **콘텐츠**: 실제 한국 공연 컨텍스트(IU 월드투어, 레미제라블 40주년, 베를린필, 조성진, SEVENTEEN, 하데스타운 등). 포스터는 실사 없이 **그라데이션 + 공연명 타이포**로 흉내.
- **카피 톤**: 운영 가능한 현실적 표현("운영시간 내 빠른 답변", "공연 당일 입장 문의 우선 응대"). 과장 금지.

## 5. 기능 요구사항 (차별화 기능 중심)

### FR-1. 공식 재판매 (핵심 차별점)
- 1:1 지정 양도를 기본 제공하지 **않는다**. 재판매는 반드시 **공식 재판매 풀**을 거친다.
- 등록 가격은 **원 티켓가의 0~10% 범위**(즉 정가의 90~100%) 내에서만. 범위 위반 시 즉시 오류 피드백.
- 구매자는 좌석번호를 **지정할 수 없고**, 등급·구역·가격범위·연석 여부만 선택 → **조건부 랜덤 매칭**.
- 매칭은 시각적 애니메이션으로 표현하고, 완료 시 **실제 좌석번호 공개 + 랜덤 시드 + 감사 원장 기록 번호** 노출.
- 수수료: 거래액 3~5% 또는 건당 1,000~2,000원. 결제 전 양도금액·수수료·최종 결제액 모두 표시.

### FR-2. 가상 티켓 vs 동적 입장 QR 분리
- 구매 직후엔 **가상 티켓**(소유 확인용, 입장 불가, 🔒 잠금 표시)만.
- 3단계 활성화 배너: ①구매 직후=가상 티켓 ②공연 1일 전=준비·기기 확인 ③공연 2~3시간 전=20초 갱신 동적 QR(앱 전용).
- 마이페이지 예매행에 **QR 상태 칩**으로 단계 표시. D-DAY 건은 "입장 QR 열기(앱·20초 갱신)" 강조 버튼.

### FR-3. 공식 양도
- 동반자 일부가 따로 입장해야 할 때만 사용. QR 직접 전달 금지 안내.
- 좌석 선택 → 받는 사람(카카오톡/이메일/휴대폰) → 0~10% 금액 → 수수료(양수자 부담) → 양도 요청.

### FR-4. 관심 공연 & 예매 알림
- 관심 공연 등록 → 예매 오픈 **D-3 + 당일** 알림. 채널: 카카오톡/앱푸시/이메일/SMS(수신 동의 토글).
- 알림 흐름 타임라인 시각화(등록 → D-3 → 당일 → 클릭 시 예매 이동).

### FR-5. 대기열
- 인기 공연 예매 진입 시 대기 인원 실시간 감소, 진행률·예상시간·처리속도, "새로고침 금지" 경고.
- 0 도달 시 "입장 차례" 상태 → 5초 카운트다운 → `booking.html` 자동 이동.

### FR-6. 운영 콘솔 (관리자)
- 사용자 페이지와 **분리된 별도 chrome**(다크 사이드바). 8개 섹션은 4.3 표 참조.
- 계정 상태(정상/주의/차단) 변경 시 확인 모달 + 토스트 + "원장 기록됨" 피드백.
- 판매 행 클릭 → 우측 폼 자동 채움 → 저장 시 확인 모달.
- 감사 원장은 **append-only 해시 체인**(SEQ·시각·PREV HASH·이벤트·검증✓)으로 표현.

### FR-7. 예매 플로우 좌석맵
- 20행(A~T)×22열, 중앙 통로(12열 위치), VIP/R/S/A 4티어 색상, 사전 매진석 표현.
- 클릭→선택/해제(최대 매수 제한), 선택 시 체크(✓)·요약 패널 실시간 갱신, 합계·수수료 자동 계산.

## 7. 인수 기준 (Acceptance Criteria)

1. 21개 HTML 파일 + 3개 CSS + 1개 JSX가 존재하고, 각 파일이 콘솔 에러 없이 단독으로 열린다.
2. 홈 → 상세 → 대기열 → 예매(3단계) → 완료 → 마이페이지 → (재판매/양도/취소/문의) 동선이 클릭만으로 끝까지 이어진다.
3. `resale.html`에서 가격 정책 범위 검증과 조건부 랜덤 매칭 애니메이션이 실제 동작한다.
4. `booking.html` 좌석맵에서 좌석 선택/해제와 요약 갱신이 실제 동작한다.
5. `confirm.html`/`mypage.html`에 가상 티켓과 동적 QR 단계가 시각적으로 구분되어 있다.
6. `admin.html` 8개 섹션이 모두 전환되고, 상태 변경/저장에 확인·토스트·원장 기록 피드백이 있다.
7. 모든 본문 텍스트 ≥ 13px, 비주얼이 미니멀·일관되며 AI slop 트로프가 없다.
8. 사용자 페이지와 관리자 콘솔의 chrome가 분리되어 있다.

2. **P1 — 대기열·로그인·취소·티켓오픈·기획전·공연장·아티스트·고객센터** (완성형 탐색·사후관리).
3. **P2 — 관리자 콘솔·공식 재판매·양도·관심공연·문의함·QR 상태 분리** (암표 통제 차별화).

각 단계 종료 시 링크 연결·콘솔 에러 점검을 수행한다.
