# Ticketground 사업계획서

## 1. 사업 개요

Ticketground는 콘서트, 뮤지컬, 스포츠, 페스티벌 티켓을 예매하고 공식 재판매까지 처리하는 클린 티켓 플랫폼이다. 기존 티켓 예매 서비스의 익숙한 흐름을 유지하면서, 플랫폼 밖 고가 양도와 계정 거래, QR 캡처 판매, 매크로 예매를 줄이는 기능을 핵심 경쟁력으로 둔다.

초기 MVP는 블록체인이나 NFT 인프라를 바로 붙이지 않고 중앙 DB, 정책 엔진, 해시 체인 감사 원장, 동적 QR 토큰으로 티켓 소유권과 거래 이력을 통제한다. 목표는 사용자가 불편하지 않게 예매하면서도, 판매자 지정 양도와 비공식 재판매가 자연스럽게 어려워지는 구조를 만드는 것이다.

## 2. 서비스 목표

Ticketground의 1차 목표는 팬과 일반 관객이 안심하고 티켓을 구매할 수 있는 공식 예매 환경을 만드는 것이다.

핵심 목표는 다음과 같다.

| 목표 | 설명 |
| --- | --- |
| 공식 예매 경험 제공 | 날짜 선택, 좌석 선택, 결제, 예매내역 확인까지 이어지는 기본 예매 흐름 제공 |
| 암표 거래 감소 | 판매자와 구매자를 직접 연결하지 않고 공식 재판매 풀을 통해 제한된 재판매만 허용 |
| 자체 플랫폼 내 양도 제한 | Ticketground에서 구매한 티켓은 Ticketground 안에서만 양도할 수 있게 하고, 양도 또는 공식 재판매 가격은 원 티켓 금액의 0~10% 범위 안에서만 조정 |
| QR 부정 사용 방지 | 구매 직후에는 가상 티켓과 내부 입장권 권한만 생성하고, 공연 임박 시점에 단계적으로 기기 확인과 실제 입장 QR 활성화 진행 |
| 관심 공연 예매 알림 | 관심 있는 공연을 캘린더에 등록하고, 예매 오픈 3일 전과 당일에 동의한 채널로 알림을 보내 예매 기회를 놓치지 않게 함 |
| 운영자 관리 기능 제공 | 판매 정보, 좌석 재고, 계정 상태, 고객 문의, 감사 원장을 관리자 페이지에서 관리 |
| 대형 서비스처럼 보이는 UX | 공연 검색, 카테고리 탐색, 상세 탭, 좌석도 팝업, 고객센터를 갖춘 완성형 예매 화면 제공 |

## 3. 사용자 및 운영자 역할

### 일반 사용자

공연, 뮤지컬, 스포츠 티켓을 예매하는 일반 구매자다. 사용자는 공연 정보를 확인하고, 개최 날짜를 선택한 뒤, 좌석 선택 창에서 좌석을 고르고 결제를 진행한다. 구매 후에는 My 예매내역에서 가상 티켓, 입장 QR 준비/활성화 상태, 공식 재판매 등록, 고객 문의를 사용할 수 있다. 또한 Ticketground 내부 정책과 가격 범위 안에서 티켓을 공식 양도하거나, 다른 사용자가 등록한 공식 양도 티켓을 양수할 수 있다.

### 판매 및 운영 관리자

공연 또는 경기 티켓을 플랫폼에 등록하고 판매 상태, 가격, 할인 예정, 오픈 예정 여부를 관리하는 사업자다. 관리자는 공연장 좌석도, 판매 좌석, 가격, 재고 상태, 계정 위험도, 고객 문의를 한 화면에서 운영할 수 있다.

### 현장 운영자

공연 당일 입장 QR을 검증하고, 이미 사용된 QR 또는 만료된 QR을 차단하는 현장 스태프다. 향후 현장 게이트 앱 또는 전용 관리자 화면과 연결할 수 있다.

## 4. 현재 MVP 구현 현황

현재 구현된 MVP는 사용자 페이지와 관리자 페이지가 분리되어 있다.

| 구분 | 현재 구현 |
| --- | --- |
| 사용자 페이지 | `http://localhost:4273` 기준 공연 탐색, 예매, 재판매, My 예매내역, 고객센터 제공 |
| 관리자 페이지 | `http://localhost:50084/admin.html` 기준 운영 콘솔 제공 |
| 서버 | Node.js 기본 `http` 서버 기반 |
| 데이터 | 로컬 JSON DB 자동 생성 및 갱신 |
| 티켓 정책 | 공식 재판매 풀, 가격 범위, 양도 횟수 제한 |
| 감사 기록 | SHA-256 기반 append-only 해시 체인 원장 |
| QR | HMAC 서명 기반 가상 티켓 QR 및 동적 입장 QR |
| 좌석도 | 잠실 올림픽 주 경기장, 블루스퀘어 등 도면형 좌석 선택 화면 |

## 5. 사용자 페이지 기능

### 5.1 공연 탐색

사용자 페이지는 첫 화면에서 대표 공연을 크게 보여주고, 공연 카테고리와 빠른 검색 기능을 제공한다.

주요 기능은 다음과 같다.

| 기능 | 설명 |
| --- | --- |
| 메인 히어로 | 대표 공연 이미지, 공연명, 예매 버튼, 내 예매내역 이동 버튼 제공 |
| 빠른 공연 검색 | 공연명, 장소, 구역 기준으로 검색 가능 |
| 카테고리 탐색 | 콘서트, 페스티벌, 뮤지컬, 스포츠로 판매 티켓 분류 |
| 판매 상태 표시 | 판매중, 오픈 예정, 할인 예정, 운영 보류 상태를 사용자에게 표시 |
| 관심 공연 등록 | 관심 있는 공연, 콘서트, 뮤지컬, 스포츠를 캘린더에 등록하고 예매 오픈 전 알림 수신 |
| 상세 페이지 이동 | 판매 티켓 클릭 시 예매 상세 화면으로 이동 |

### 5.2 관심 공연 캘린더 및 예매 알림

사용자는 관심 있는 공연, 콘서트, 뮤지컬, 스포츠 경기, 페스티벌을 개인 캘린더에 등록할 수 있다. 등록된 관심 공연은 My 예매내역 또는 별도 관심 공연 화면에서 확인하고, 예매 오픈일이 가까워지면 사전 알림을 받을 수 있다.

추천 기능 구성은 다음과 같다.

| 기능 | 설명 |
| --- | --- |
| 관심 공연 등록 | 공연 상세, 검색 결과, 카테고리 목록에서 관심 공연으로 저장 |
| 캘린더 보기 | 관심 공연의 예매 오픈일, 공연일, 장소, 판매 상태를 월간/목록 형태로 확인 |
| 예매 D-3 알림 | 티켓 예매 오픈일 약 3일 전 카카오톡, 문자, 앱 푸시, 이메일 중 동의한 채널로 알림 |
| 당일 재알림 | 예매 오픈 당일 또는 오픈 1시간 전 앱 푸시/카카오톡 재알림 |
| 상태 변경 알림 | 오픈 예정, 할인 예정, 판매중, 매진 임박 상태 변경 시 알림 |
| 수신 설정 | 공연별 알림 켜기/끄기, 채널별 수신 동의, 알림 해제 제공 |

알림 흐름은 다음과 같다.

```text
관심 공연 등록
→ 예매 오픈일 계산
→ D-3 사전 알림 예약
→ 오픈 당일 재알림
→ 사용자가 알림 클릭
→ 예매 상세 페이지 또는 앱 예매 화면으로 이동
```

이 기능은 사용자가 예매일을 놓치지 않도록 돕고, 플랫폼 입장에서는 재방문율과 예매 전환율을 높이는 역할을 한다. 다만 알림은 사용자가 직접 관심 공연으로 등록했거나 수신에 동의한 채널에만 발송하며, 수신 해제 기능을 명확히 제공한다.

### 5.3 상품 상세 탭

예매 상세 화면에는 티켓 구매 전에 확인해야 하는 정보를 탭 형태로 제공한다.

| 탭 | 제공 정보 |
| --- | --- |
| 공지사항 | 예매 전 필수 확인 사항, 공식 거래 안내 |
| 캐스팅 | 공연 출연진 또는 구성 정보 |
| 상품상세 | 공연 설명, 무대 구성, 관람 특징 |
| 가격/할인 | 각 좌석 가격 및 할인 정보 |
| 이용안내 | 공연 날짜, 공연 시간, 관람 연령, 티켓 수령 및 입장 안내 |
| 취소/환불 | 취소 가능 시점과 환불 규정 |
| 장소 | 공연장명과 주소 |

각 탭은 클릭 시 메인 화면으로 이탈하지 않고 해당 상세 영역만 보여주는 방식으로 정리되어 있다.

### 5.4 예매 흐름

사용자 예매 흐름은 다음 순서로 구성된다.

```text
공연 선택
→ 예매 날짜 선택
→ 공연 시간 선택
→ 좌석 선택 창 열기
→ 좌석 선택
→ 결제
→ My 예매내역 확인
```

예매 날짜는 관리자가 등록한 실제 개최 날짜만 선택할 수 있고, 공연 시간이 여러 개라면 사용자가 원하는 회차 시간을 선택한 뒤 좌석 선택 창을 연다. 오픈 예정, 할인 예정, 운영 보류 상태의 티켓은 사용자가 예매할 수 없고 상태 문구만 확인할 수 있다.

### 5.5 좌석 선택

좌석 선택은 예매 상세 페이지 안에 모든 기능을 길게 노출하지 않고, 좌석 선택 버튼을 누르면 별도 팝업형 좌석 선택 창으로 진행되도록 구성되어 있다.

현재 구현된 좌석 선택 기능은 다음과 같다.

| 기능 | 설명 |
| --- | --- |
| 팝업형 좌석 선택 | PC에서는 작은 팝업 창으로 좌석 선택 화면을 열어 기존 예매 페이지를 유지 |
| 모바일 대응 | 좁은 화면에서는 전체 페이지 이동 방식으로 좌석 선택 가능 |
| 공연 정보 전달 | 공연명, 장소, 일시, 최대 구매 매수를 URL 파라미터로 전달 |
| 잠실 올림픽 주 경기장 도면 | 경기장형 원형 좌석 블록과 구역별 상세 좌석 선택 제공 |
| 블루스퀘어 도면 | 1층, 2층, 3층 도면 기반 층별 구역 선택 제공 |
| 좌석 수량 제한 | `maxTickets` 기준 최대 선택 매수 제한 |
| 선택 좌석 요약 | 선택한 구역, 열, 번호, 가격을 우측 패널에 표시 |

1인이 여러 장의 티켓을 구매한 경우에는 대표 구매자 중심 입장 정책을 기본으로 한다. 대표 구매자 앱에서 구매한 티켓의 입장 QR을 관리하고, 동반자는 대표 구매자와 함께 입장한다. 티켓별로 다른 사람에게 따로 넘겨야 하는 경우에는 플랫폼 내 공식 양도 기능을 사용해야 한다.

### 5.6 결제 기능

현재 MVP는 실제 PG 결제 전 단계의 시뮬레이션 구조다.

지원 결제수단은 다음과 같다.

| 결제수단 | 현재 처리 방식 |
| --- | --- |
| 충전금 | 데모 잔액에서 티켓 금액 차감. 실제 초기 서비스에서는 기본 결제수단이 아니라 추후 선택 옵션으로 검토 |
| 신용카드 | 결제 승인 완료 상태로 시뮬레이션 |
| 계좌이체 | 결제 승인 완료 상태로 시뮬레이션 |
| 무통장 입금 | 입금대기 상태로 예매 처리 |
| 휴대폰 결제 | 결제 승인 완료 상태로 시뮬레이션 |

실제 서비스의 초기 결제는 카드, 간편결제, 계좌이체 등 일반 PG 결제를 기본으로 하고, 충전금은 빠른 매칭이나 팬 월렛이 필요한 고도화 단계에서 별도 검토한다. PG사 연동 시에는 결제 승인, 취소, 환불, 정산, 세금계산서, 에스크로 또는 예치금 분리 보관 정책이 필요하다.

### 5.7 My 예매내역

My 예매내역은 구매한 티켓과 입장권 관련 기능을 모아둔 영역이다.

주요 기능은 다음과 같다.

| 기능 | 설명 |
| --- | --- |
| 보유 티켓 목록 | 구매한 티켓의 공연명, 일시, 장소, 좌석, 상태 표시 |
| 가상 티켓 보기 | 구매 직후 확인 가능한 예매 확인용 가상 티켓 제공 |
| 입장 QR 활성화 | 공연 1일 전 준비 상태를 안내하고, 공연 2~3시간 전부터 모바일에서만 실제 게이트 통과 가능한 동적 입장 QR 활성화 |
| 대표자 동반 입장 | 1인이 여러 장을 구매한 경우 대표 구매자 앱에서 전체 QR을 관리하고 동반자는 함께 입장 |
| 공식 재판매 등록 | 보유 티켓을 가격 정책 안에서 공식 재판매 풀에 등록 |
| 회원정보 수정 | 데모 기준 닉네임 수정 기능 제공 |

가상 티켓은 실제 입장권이 아니라 구매 사실을 보여주는 확인용 티켓이다. 실제 게이트 통과가 가능한 동적 입장 QR은 10장의 QR 정책에 따라 공연 2~3시간 전부터 모바일에서만 활성화한다. 다매 구매 티켓을 각자 따로 사용해야 한다면 개별 QR을 임의 전달하지 않고 공식 양도 절차로 소유권을 이전해야 한다.

### 5.8 고객센터

사용자 페이지에는 고객센터 메뉴와 우측 하단 1:1 문의 위젯이 있다. 초기에는 1인 운영을 전제로 하므로 자체 실시간 채팅 앱을 먼저 만들기보다, 플랫폼 내부 문의함과 카카오톡 채널 상담을 병행한다.

현재 구현된 기능은 다음과 같다.

| 기능 | 설명 |
| --- | --- |
| 문의 생성 | 사용자가 제목과 메시지를 입력해 문의 접수 |
| 대화형 메시지 | 사용자와 상담원 메시지를 스레드 형태로 표시 |
| 상태 표시 | 답변 대기, 답변 완료, 상담 종료 상태 표시 |
| 자동 갱신 | 사용자 화면에서 문의 내역을 주기적으로 갱신 |
| 관리자 연동 | 관리자 페이지 고객 문의 탭에서 같은 문의를 확인하고 답변 가능 |

초기 고객센터 운영 정책은 다음과 같다.

| 구분 | 정책 |
| --- | --- |
| 기본 문의 | 로그인 기반 플랫폼 내부 문의함에서 접수 |
| 빠른 상담 | 카카오톡 채널 1:1 채팅으로 연결 |
| 긴급 문의 | 입장 QR, 결제 실패, 공연 당일 입장 문제를 우선 처리 |
| 응대 표현 | 24시간 실시간 상담으로 안내하지 않고 운영시간 내 빠른 상담으로 안내 |
| 문의 연결 | 문의 시 예매번호, 공연명, 티켓 상태를 자동으로 함께 표시 |
| FAQ | 환불, 양도, 입장 QR, 앱 설치, 결제 오류를 자주 묻는 질문으로 제공 |

향후에는 문의량이 늘어나는 시점에 카카오 상담톡, 이메일 알림, 상담 배정, 응답 시간 관리 기능을 붙인다. 자체 실시간 채팅 앱은 초기 MVP에서는 만들지 않고, 전용앱 안에는 고객센터 진입 버튼과 문의 내역 확인 기능만 둔다.

## 6. 공식 재판매 기능

Ticketground의 핵심 차별점은 1:1 지정 양도를 기본 기능으로 제공하지 않는 것이다. 사용자가 티켓을 다시 판매하고 싶으면 공식 재판매 풀에 등록해야 한다.

### 6.1 재판매 등록

보유 티켓 소유자는 My 예매내역에서 티켓을 공식 재판매 풀에 등록할 수 있다. 등록 가격은 티켓별 최소 가격과 최대 가격 사이에서만 가능하다.

```text
보유 티켓 선택
→ 판매 가격 입력
→ 가격 정책 검증
→ 공식 재판매 풀 생성
→ 티켓 상태 IN_RESALE_POOL 전환
```

### 6.2 대기자 풀

구매자는 특정 판매자 또는 특정 좌석번호를 지정하지 않고 공식 재판매 풀에 대기 신청한다. 다만 사용자 불편을 줄이기 위해 원하는 좌석 등급, 구역, 가격 범위, 연석 필요 여부는 선택할 수 있다.

```text
공식 재판매 티켓 확인
→ 좌석 등급/구역/가격 범위/연석 조건 선택
→ 대기 신청
→ 대기자 풀 등록
→ 운영자 또는 시스템 매칭
```

### 6.3 조건부 랜덤 매칭

재판매 풀에 대기자가 있으면 구매자가 선택한 조건 안에서 랜덤 매칭을 진행한다. 일반 예매는 사용자가 원하는 지정석을 직접 선택할 수 있지만, 공식 재판매와 양도는 외부에서 약속한 특정 좌석 지정 거래를 막기 위해 조건부 랜덤 매칭을 기본으로 한다.

매칭 조건은 다음과 같다.

| 조건 | 설명 |
| --- | --- |
| 좌석 등급 | VIP석, R석, S석, A석 등 사용자가 원하는 좌석 등급 |
| 구역 또는 층 | 1층, 2층, 중앙, 측면 등 관리자가 허용한 범위 |
| 가격 범위 | 사용자가 부담 가능한 최대 가격과 플랫폼 가격 정책 |
| 연석 필요 여부 | 2매 이상 구매 시 연석만 매칭할지 여부 |
| 매수 | 사용자가 구매하려는 티켓 수량 |

매칭이 완료되면 실제 좌석번호를 공개하고, 양수자는 카드, 간편결제, 계좌이체 등 일반 PG 결제로 티켓 양도 금액과 공식 양도 수수료를 결제한다. 결제가 승인되면 티켓 소유자를 양수자로 변경하고, 양도자에게는 정산 계좌로 판매금액을 지급한다.

수수료 처리 원칙은 다음과 같다.

| 항목 | 정책 |
| --- | --- |
| 결제 방식 | 초기/베타는 카드, 간편결제, 계좌이체 등 일반 PG 결제 |
| 양수자 부담 | 티켓 양도 금액 + 공식 양도 수수료 |
| 양도자 수령 | 판매 확정 후 정산 계좌로 지급 |
| 플랫폼 수익 | 양도 완료 시점에 공식 양도 수수료 수취 |
| 권장 수수료 | 거래 금액의 3~5% 또는 건당 1,000~2,000원 |
| 충전금 | 초기 필수 기능에서 제외하고, 추후 빠른 매칭/빠른 정산/팬 월렛 옵션으로 검토 |

양도 수수료와 정산 수수료의 세부 기준은 12장 양도 결제, 정산 및 충전금 관리 정책을 따른다. 사용자가 매칭을 확정하기 전에는 티켓 양도 금액, 공식 양도 수수료, 최종 결제 금액을 모두 표시해야 한다.

이 구조는 다음 효과를 목표로 한다.

| 효과 | 설명 |
| --- | --- |
| 지정 양도 억제 | 판매자와 구매자가 외부에서 특정 좌석을 약속해도 그 좌석이 해당 구매자에게 넘어간다는 보장을 없앰 |
| 사용자 선택권 보존 | 좌석번호 직접 선택은 막지만 좌석 등급, 구역, 가격 범위, 연석 여부는 선택 가능 |
| 가격 정책 적용 | 과도한 재판매 가격을 시스템에서 제한 |
| 거래 이력 보존 | 재판매 등록, 대기 신청 조건, 매칭 결과, 랜덤 시드를 감사 원장에 기록 |
| 공식 거래 유도 | 외부 거래보다 플랫폼 안 재판매가 더 편하도록 설계 |

## 7. 관리자 페이지 기능

관리자 페이지는 사용자 페이지와 분리된 운영 콘솔이다. 현재 기준 접속 주소는 `http://localhost:50084/admin.html`이다.

### 7.1 운영 현황

관리자는 판매중 좌석, 공식 재판매 풀, 고객 문의, 주의 계정, 감사 원장 상태를 한 화면에서 확인할 수 있다.

| 지표 | 설명 |
| --- | --- |
| 판매중 좌석 | 전체 좌석 중 예매 가능한 좌석 수 |
| 공식 재판매 풀 | 열려 있는 재판매 풀 수 |
| 고객 문의 | 답변 대기 또는 진행 중 문의 수 |
| 주의 계정 | WATCHLIST 또는 신뢰 점수 50점 미만 계정 수 |
| 감사 원장 | 원장 기록 수와 해시 체인 검증 상태 |

### 7.2 계정 관리

관리자는 계정별 상태를 선택형 옵션으로 변경할 수 있다.

지원 상태는 다음과 같다.

| 상태 | 의미 |
| --- | --- |
| 정상 | 일반 거래 가능 계정 |
| 주의 | 추가 모니터링이 필요한 계정 |
| 차단 | 거래 제한 계정 |

계정 상태 변경은 우측 상단 수정 버튼을 눌러 저장하며, 저장 전 “수정하시겠습니까?” 확인 팝업을 거친다. 변경 내역은 제재 이력과 감사 원장에 기록된다.

### 7.3 암표 대응

운영자는 지정 양도 차단 시뮬레이션 기능을 통해 장외 거래 시도를 기록할 수 있다.

처리 방식은 다음과 같다.

```text
시도 계정 선택
→ 대상 계정 선택
→ 티켓 선택
→ 제안 가격 입력
→ 차단 기록 남기기
→ 신뢰 점수 차감
→ 감사 원장 기록
```

현재 정책상 지정 양도 시도는 신뢰 점수 차감과 WATCHLIST 전환 근거로 사용된다.

### 7.4 티켓 판매 관리

관리자는 공연 판매 정보를 등록하거나 수정할 수 있다.

| 항목 | 설명 |
| --- | --- |
| 행사 | 수정 대상 공연 선택 |
| 행사명 | 사용자 페이지에 노출되는 공연명 |
| 행사 유형 | 콘서트, 뮤지컬, 페스티벌, 스포츠 |
| 판매 상태 | 판매중, 오픈 예정, 할인 예정, 운영 보류 |
| 사용자 노출 문구 | 판매 상태와 함께 사용자 페이지에 표시될 안내 문구 |
| 할인율 | 할인 예정 또는 프로모션 표시에 활용 |
| 개최 날짜 및 시간 | 예매 가능한 실제 공연 날짜 |
| 개최 장소 | 좌석도 API와 연결되는 공연장 |
| 구역별 가격 | 공연장별 좌석 등급과 구역에 맞춘 가격 |

판매 정보 저장 시 사용자 페이지에도 즉시 반영된다. 미판매 좌석의 가격은 새 구역별 가격에 맞춰 동기화된다.

### 7.5 좌석도 API 미리보기

관리자는 공연장을 선택하면 해당 장소의 좌석도 API 정보를 확인할 수 있다.

현재 지원되는 공연장 예시는 다음과 같다.

| 공연장 | 좌석도 |
| --- | --- |
| 잠실 올림픽 주 경기장 | 대형 경기장형 좌석 배치 |
| KSPO Dome | 실내 공연장형 좌석 배치 |
| 난지한강공원 | 야외 페스티벌형 배치 |
| 블루스퀘어 | 1층, 2층, 3층 극장형 좌석 배치 |

관리자 페이지의 좌석도 선택은 사용자 예매 페이지의 좌석 선택 API와 연결된다.

### 7.6 좌석 재고 관리

관리자는 아직 소유자가 없는 좌석을 판매중 또는 운영 보류 상태로 전환할 수 있다. 이미 소유자가 있거나 재판매 상태인 좌석은 운영자가 임의로 재고 상태를 바꾸지 못하게 제한한다.

### 7.7 고객 문의 관리

관리자 페이지의 고객 문의 탭에서는 사용자가 남긴 1:1 문의를 확인하고 답변할 수 있다. 1인 운영자는 모든 문의를 직접 처리해야 하므로, 문의 대기열은 긴급도와 예매 상태를 기준으로 정렬되어야 한다.

| 기능 | 설명 |
| --- | --- |
| 문의 대기열 | 최근 문의를 최신순으로 표시 |
| 상담 상세 | 선택한 문의의 전체 대화 내역 표시 |
| 답변 보내기 | 상담원 메시지를 사용자 문의 스레드에 추가 |
| 상태 변경 | 답변완료 또는 상담종료로 상태 변경 |
| 원장 기록 | 문의 생성, 답변, 상태 변경 내역 기록 |
| 긴급도 표시 | 입장 QR, 결제, 공연 당일 문의를 우선 표시 |
| 카카오 연결 | 사용자가 카카오톡 채널 상담을 원하면 채널 상담 링크 제공 |

초기에는 문의가 들어오면 운영자에게 카카오톡 또는 문자 알림을 보내고, 운영자는 관리자 페이지에서 답변한다. 카카오톡 채널로 들어온 상담은 채널 관리자센터에서 응대하되, 예매번호가 필요한 민감 문의는 플랫폼 내부 문의함으로 유도한다. 상담량이 증가하면 카카오 상담톡 또는 CS SaaS를 붙여 상담 배정, 태그, 자동응답, 종료 사유, 응답 시간 지표를 관리한다.

## 8. 백엔드 및 API 구성

현재 서버는 Node.js 기본 `http` 모듈로 동작한다. 사용자 페이지와 관리자 페이지는 같은 서버 코드에서 각각 다른 포트로 제공된다.

```text
사용자 페이지: PORT 4273
관리자 페이지: ADMIN_PORT 50084
```

현재 주요 API는 다음과 같다.

| API | 용도 |
| --- | --- |
| `GET /api/state` | 사용자 페이지 상태 조회 |
| `GET /api/users/:userId/watchlist` | 사용자의 관심 공연 및 캘린더 조회 |
| `GET /api/admin/summary` | 관리자 운영 현황 조회 |
| `GET /api/admin/venues` | 관리자 공연장 및 행사 정보 조회 |
| `GET /api/seat-map` | 공연장 좌석도 API 조회 |
| `GET /api/events/:eventId/seat-map` | 특정 행사 좌석도 조회 |
| `POST /api/tickets/buy` | 1차 티켓 구매 |
| `POST /api/watchlist` | 관심 공연 등록, 알림 채널, 캘린더 표시 설정 |
| `POST /api/watchlist/notify` | 예매 오픈 D-3, 당일, 상태 변경 알림 예약 또는 발송 |
| `POST /api/tickets/virtual-qr` | 예매 확인용 가상 티켓 발급 |
| `POST /api/tickets/qr` | 활성화된 입장권 권한에 대한 20초짜리 동적 QR 토큰 발급 |
| `POST /api/gate/verify` | 현장 게이트 QR 검증 |
| `POST /api/resale/list` | 공식 재판매 등록 |
| `POST /api/resale/join` | 공식 재판매 대기 신청 |
| `POST /api/resale/draw` | 공식 재판매 조건부 랜덤 매칭 |
| `POST /api/security/direct-transfer-attempt` | 지정 양도 차단 기록 |
| `POST /api/admin/events/sale` | 관리자 판매 정보 저장 |
| `POST /api/admin/users/statuses` | 계정 상태 일괄 수정 |
| `POST /api/admin/tickets/status` | 좌석 재고 상태 변경 |
| `POST /api/support/threads` | 사용자 문의 생성 |
| `POST /api/admin/support/messages` | 관리자 문의 답변 |

## 9. 데이터 및 감사 원장

현재 MVP는 로컬 JSON DB로 사용자, 공연, 공연장, 티켓, 재판매 풀, 고객 문의, 감사 원장을 관리한다.

주요 데이터는 다음 구조로 나뉜다.

| 데이터 | 역할 |
| --- | --- |
| users | 회원, 충전금/보유금 옵션, 계정 상태, 신뢰 점수, 제재 이력 |
| venues | 공연장, 주소, 좌석도 메타데이터 |
| events | 행사명, 카테고리, 날짜, 판매 상태, 구역별 가격 |
| watchlist | 사용자별 관심 공연, 캘린더 표시, 알림 채널, 수신 설정 |
| notificationJobs | 예매 오픈 D-3, 당일, 상태 변경 알림 예약과 발송 이력 |
| tickets | 좌석별 티켓, 소유자, 상태, 가격, 양도 횟수 |
| resalePools | 공식 재판매 풀, 판매자, 대기자, 매칭 상태 |
| supportThreads | 사용자 문의와 상담원 답변 |
| ledger | 주요 이벤트의 해시 체인 기록 |

감사 원장은 각 기록이 이전 기록의 해시를 참조하는 append-only 구조다. 이를 통해 주요 이벤트의 사후 변경 여부를 탐지할 수 있다.

기록 대상 예시는 다음과 같다.

| 이벤트 | 기록 내용 |
| --- | --- |
| 최초 데이터 생성 | 공연, 공연장, 좌석 발행 스냅샷 |
| 티켓 구매 | 구매자, 티켓, 공연, 좌석, 결제수단 |
| 공식 재판매 등록 | 판매자, 티켓, 가격, 정책 |
| 재판매 매칭 | 판매자, 구매자, 대기 신청 조건, 가격, 수수료, 랜덤 시드, 매칭 결과 |
| 지정 양도 차단 | 시도 계정, 대상 계정, 티켓, 제안 가격 |
| QR 토큰 발급/검증 | 티켓, 만료 시각, 검증 결과 |
| 관심 공연/알림 | 관심 등록, 알림 예약, 발송 결과, 수신 해제 |
| 계정 상태 변경 | 대상 계정, 변경 상태, 사유 |
| 고객 문의 | 문의 생성, 답변, 상태 변경 |

## 10. 입장 QR 및 가상 티켓 정책

입장 QR 정책은 `입장qr 문제점.md`의 설계를 기준으로 한다.

### 10.1 기본 원칙

구매 중에는 예매 속도가 중요하므로 일반 사용자에게 OTP를 기본 요구하지 않는다. 구매 직후 사용자에게 실제 입장 QR을 바로 제공하지 않고, 예매 확인용 가상 티켓과 서버 내부 입장권 권한을 생성한다. 위험 계정이거나 새 기기, 다계정, 비정상 구매 정황이 있으면 구매 직후 OTP로 소유자를 확인하고, 공연 1일 전에는 준비 상태와 기기 확인 안내만 활성화한다. 실제 게이트 통과가 가능한 동적 입장 QR은 공연 2~3시간 전부터 활성화한다.

```text
구매 직후: 가상 티켓 + 내부 입장권 권한 생성, 필요 시 OTP 소유자 확인
공연 1일 전: 입장 QR 준비 상태, 기기 확인, 안내 활성화
공연 2~3시간 전: 실제 게이트 통과 가능한 동적 QR 활성화
입장 시: 20초마다 QR 갱신
게이트: 1회 검증 후 사용 처리 및 폐기
```

### 10.2 가상 티켓

가상 티켓은 입장권이 아니라 소유권 확인용 티켓이다.

가상 티켓의 역할은 다음과 같다.

| 역할 | 설명 |
| --- | --- |
| 구매 확인 | 사용자가 예매 성공 여부를 확인 |
| 소유권 표시 | 현재 티켓 소유자와 좌석 정보 표시 |
| 내부 권한 연결 | 서버 내부 입장권 권한과 연결되지만 QR 자체는 입장에 사용 불가 |
| 재판매 준비 | 공식 재판매 또는 양도 가능 상태 확인 |
| 실제 입장 차단 | 가상 티켓 QR은 게이트 입장 검증에 사용할 수 없음 |

### 10.3 입장 QR

입장 QR은 고정 이미지를 미리 발급하는 방식이 아니라, 활성화 시점 이후 사용자가 QR 화면을 열 때마다 생성되는 20초짜리 HMAC 서명 기반 토큰으로 설계한다.

QR payload에는 개인정보를 넣지 않고, 외부에서 사용자를 알아볼 수 없도록 랜덤 토큰과 서명만 사용한다.

```text
ticket_token
jti
exp
signature
```

서버 내부에서는 다음 매핑을 보관한다.

```text
jti
→ admission_credential_id
→ ticket_id
→ booking_id
→ user_id
→ device_id
→ token_issued_at
```

외부에서는 사용자를 식별할 수 없지만, 부정 거래나 분쟁이 발생하면 서버 내부 로그로 어느 티켓에서 나온 QR인지 추적할 수 있다.

### 10.4 QR 보안 정책

| 정책 | 설명 |
| --- | --- |
| 20초 만료 | 캡처 이미지가 빠르게 무효화되도록 함 |
| 1회 사용 | 먼저 검증된 1회만 입장 처리 |
| 단계별 활성화 | 구매 직후 권한 생성과 필요 시 OTP 확인, 1일 전 준비 안내, 2~3시간 전 실제 입장 QR 활성화 |
| 동적 워터마크 | 화면에 내부 추적 코드와 시간 정보를 표시 |
| 발급 로그 저장 | QR 열람 시 사용자, 티켓, 기기, IP, 시각 기록 |
| 리스크 탐지 | 여러 기기 또는 여러 IP에서 반복 열람 시 추가 인증 |
| 공식 재판매 유도 | 캡처 판매보다 플랫폼 내 재판매가 편하도록 설계 |

### 10.5 활성화 시점

이 절은 10.1의 기본 원칙을 실제 운영 절차로 풀어쓴다. 핵심은 구매 중 예매 속도를 해치지 않고, 구매 직후부터 QR 활성화 전까지 위험 기반 인증을 분산하는 것이다.

추천 운영 방식은 다음과 같다.

```text
구매 직후: admission_credential 생성, 가상 티켓 표시, 위험 계정은 OTP 확인
공연 1일 전: QR 준비 상태, 기기 확인, 안내 문구 활성화
공연 2~3시간 전: 실제 입장 QR 버튼 활성화
QR 토큰 생성: 사용자가 버튼을 누를 때 실시간 발급
QR 갱신: 20초마다 재발급
위험 계정: 당일 추가 인증
서버 처리: Redis 기반 단기 토큰 캐시
```

이 방식은 사용자의 준비 불안을 줄이면서도 실제 입장 가능한 QR의 노출 시간을 짧게 유지한다. 구매 직후 고정 QR 이미지를 만들어두는 방식은 캡처 판매와 계정 양도 리스크가 커지므로 사용하지 않는다.

### 10.6 입장 QR 표시 채널

가상 티켓은 웹과 앱에서 모두 확인할 수 있어도 된다. 가상 티켓은 구매 사실과 소유 상태를 보여주는 확인용 정보이며, 실제 입장 처리를 하지 않기 때문이다.

반대로 실제 게이트 통과가 가능한 입장 QR은 전용앱에서만 표시하는 것을 기본 정책으로 한다. 카카오톡이나 문자는 QR 이미지를 전달하는 수단이 아니라, 사용자를 전용앱으로 진입시키는 알림 수단으로 사용한다.

추천 운영 흐름은 다음과 같다.

```text
구매 직후:
웹/앱에서 가상 티켓 확인 가능
위험 계정이면 OTP 확인 후 입장권 권한 정상 유지

공연 1일 전:
카카오톡/문자 발송
"입장 QR 준비 상태를 확인해주세요"
웹에서는 준비 상태와 앱 설치 안내만 표시

공연 2~3시간 전:
카카오톡/문자 재발송
"입장 QR이 활성화되었습니다"
링크 클릭 시 앱 딥링크 실행

앱에서:
로그인/기기 확인
위험 계정이면 OTP 또는 고객센터 확인
정상 계정이면 20초 갱신 QR 표시

현장:
QR 1회 사용 후 폐기
```

채널별 역할은 다음과 같이 구분한다.

| 채널 | 역할 |
| --- | --- |
| 웹 | 가상 티켓, 예매내역, 양도/재판매, QR 준비 상태와 앱 설치 안내 |
| 전용앱 | 실제 입장 QR 표시, 20초 갱신, 기기 확인, 위험 계정 추가 인증 |
| 카카오톡/문자 | QR 직접 전달이 아니라 공연 1일 전 준비 안내와 2~3시간 전 앱 딥링크 알림 |
| 현장 고객센터 | 앱 사용 불가, 배터리 방전, 장애 상황의 본인확인 및 예외 입장 처리 |

1인 다매 구매 시에는 대표 구매자 앱에서 전체 티켓의 QR을 표시하는 것을 기본으로 한다. 동반자는 대표 구매자와 함께 입장하고, 일부 티켓만 다른 사람에게 넘겨야 하는 경우에는 공연 전 공식 양도 기능으로 소유권을 이전한다. 이렇게 하면 암표상이 한 계정으로 여러 장을 구매한 뒤 여러 사람에게 QR을 쪼개 전달하는 비용과 리스크가 커진다.

웹 QR 예외 발급은 기본적으로 금지하되, 운영상 불가피한 경우에만 제한적으로 허용한다.

| 예외 상황 | 처리 |
| --- | --- |
| 앱 설치 불가 기기 | 현장 고객센터 부스에서 본인확인 후 1회용 입장 처리 |
| 배터리 방전 | 신분증, 예매정보, 본인확인 후 현장 처리 |
| 앱 장애 | 운영자가 전체 장애 모드로 전환해 웹 임시 QR 또는 게이트 수기 확인 |
| 해외 사용자 | 앱 설치 안내를 사전 발송하고, 실패 시 현장 확인 플로우 제공 |

이 채널 구분을 기준으로 15장의 카카오톡, 문자, 앱 딥링크 연동을 설계한다.

### 10.7 전용앱 생체인증 및 기기 키 정책

티켓 구매 후 또는 입장 QR 활성화 시점에는 전용앱에서 생체인증 또는 기기 잠금 인증으로 신뢰 기기 기반 사용자 확인을 수행한다. 다만 플랫폼은 사용자의 얼굴, 지문 등 생체정보 원본을 수집하거나 저장하지 않는다. 생체인증은 OS가 처리하고, Ticketground 앱과 서버는 인증 성공 여부와 앱 내부 기기 키 서명 결과만 사용한다.

권장 인증 구조는 다음과 같다.

```text
티켓 구매 직후:
위험 신호가 있으면 OS 생체인증 또는 기기 잠금 인증 요청
앱 내부 기기 키로 서버 요청 서명
서버는 신뢰 기기와 서명 결과 확인

입장 QR 활성화 시:
OS 생체인증 또는 기기 잠금 인증 요청
앱 내부 기기 키로 QR 발급 요청 서명
서버는 신뢰 기기, 서명 결과, 위험 점수 확인 후 QR 발급
```

생체정보 변경, 기기 잠금 해제 방식 변경, 앱 재설치, 기기 보안 설정 변경으로 앱 내부 기기 키가 무효화되면 QR 발급을 바로 허용하지 않는다. 이 경우 OTP 또는 고객센터 확인 후 신뢰 기기를 재등록한다.

| 항목 | 정책 |
| --- | --- |
| 생체정보 원본 | 플랫폼에서 수집하거나 저장하지 않음 |
| 생체인증 처리 | iOS, Android 등 기기 OS가 처리 |
| 앱 서버 확인값 | OS 인증 성공 여부, 앱 내부 기기 키 서명 결과, 신뢰 기기 상태 |
| 키 무효화 상황 | 생체정보 변경, 기기 잠금 방식 변경, 앱 재설치, 기기 보안 설정 변경 |
| 복구 절차 | OTP 또는 고객센터 확인 후 신뢰 기기 재등록 |

이 정책의 목적은 생체정보 자체를 본인확인 수단으로 보관하는 것이 아니라, 공연 임박 시점에 등록된 신뢰 기기에서 실제 사용자가 접근했는지 확인하는 것이다.

### 10.8 캡처 방지 한계

모바일 웹이나 앱에서 화면 캡처를 제한해도 다른 휴대폰으로 화면을 촬영하는 행위는 기술적으로 완전히 막을 수 없다.

따라서 QR 정책은 다음 세 가지를 함께 적용해야 한다.

```text
방지: 캡처 방지, 짧은 만료, 1회 사용
추적: 비개인 내부 추적 코드, 발급 로그, 워터마크
무력화: 재사용 차단, 위험 계정 보류, 공식 재판매 유도
```

## 11. 계정 양도 및 매크로 대응

암표상은 QR 캡처뿐 아니라 계정 자체를 넘기는 방식으로 거래할 수 있다. 따라서 Ticketground는 계정 로그인과 입장권 사용 권한을 분리해서 봐야 한다.

### 11.1 위험 기반 인증

모든 사용자에게 강한 인증을 요구하면 정상 구매자가 불편해진다. 따라서 기본 사용자는 편하게 두고 위험 신호가 있을 때만 추가 확인을 요구한다.

| 상황 | 처리 |
| --- | --- |
| 평소 사용하던 기기 | QR 활성화 시간 이후 바로 열람 |
| 새 기기 로그인 | OTP 1회 |
| 구매 직후 위험 신호 발생 | OTP 1회 또는 입장권 권한 임시 보류 |
| 공연 1일 전 이후 비밀번호 변경 | 추가 확인 |
| 공연 당일 새 기기와 비밀번호 변경 동시 발생 | QR 보류 또는 고객센터 확인 |
| 공식 양도 완료 | 새 소유자에게 정상 권한 이전 |

### 11.2 리스크 점수

위험 점수는 계정 상태, 기기, IP, 결제수단, QR 열람 패턴, 구매 속도, 제재 이력 등을 조합해 계산한다.

```text
risk_score < 30: 정상
30~60: OTP 1회
60~80: QR 열람 지연 또는 고객센터 확인
80 이상: QR 보류 또는 구매 제한
```

### 11.3 분산형 매크로 계정 대응

계정 하나당 티켓 하나씩 구매하는 방식은 단순 계정 제한으로 막기 어렵다. 따라서 사람, 기기, 결제수단, 네트워크, 행동 패턴을 묶어서 봐야 한다.

주요 탐지 신호는 다음과 같다.

| 신호 | 설명 |
| --- | --- |
| 본인인증 중복 | 같은 본인인증 정보에서 다수 계정 생성 |
| 결제수단 fingerprint | 같은 카드 또는 간편결제 식별값으로 다수 계정 구매 |
| 기기 fingerprint | 같은 기기에서 여러 계정 로그인 |
| IP/ASN 패턴 | 프록시, VPN, 데이터센터 IP 대역의 반복 구매 |
| 행동 패턴 | 비정상적으로 빠른 좌석 선택, 마우스/터치 이벤트 부족 |
| 결제 실패 반복 | 좌석 선점 후 결제 실패를 반복하는 패턴 |

목표는 매크로를 완전히 없애는 것이 아니라 매크로 운영 비용을 높이고, 공식 재판매가 더 안전하고 편한 경로가 되도록 만드는 것이다.

### 11.4 매크로 예매 제재 정책

Ticketground는 매크로, 자동화 도구, 비정상 대량 요청, 대기열 우회, 다계정 조직 구매를 약관상 금지한다. 의심 신호가 감지되면 즉시 영구 차단하기보다 추가 인증, 구매 보류, 입장 QR 활성화 보류, 공식 재판매 제한, 계정 상태 변경을 단계적으로 적용한다.

제재의 기본 원칙은 다음과 같다.

```text
탐지 = 자동화
확정 제재 = 운영자 검토 + 로그 증빙
정상 사용자 = 추가 인증으로 통과
고위험 사용자 = QR 보류/구매 보류
확정 암표상 = 취소/차단/신고 검토
```

단계별 제재 기준은 다음과 같다.

| 단계 | 상황 | 조치 |
| --- | --- | --- |
| 1단계 | 비정상 속도, 반복 요청, 좌석 선점 반복 | 대기열 재검증, CAPTCHA, OTP |
| 2단계 | 같은 기기, IP, 결제수단으로 다계정 구매 의심 | 구매 수량 제한, 좌석 선점 시간 단축, 구매 직후 OTP 또는 입장권 권한 임시 보류 |
| 3단계 | 명확한 매크로 패턴 또는 우회 시도 | 예매 보류, 입장 QR 활성화 보류, 고객센터 확인 |
| 4단계 | 매크로 구매 후 재판매 정황 | 티켓 취소, 계정 주의/차단, 재판매 제한, 증빙 보관 |
| 5단계 | 반복적 또는 조직적 암표 행위 | 계정 영구 차단, 결제수단 제한, 관계기관 신고 검토 |

매크로 제재 판단에 활용하는 증빙은 다음과 같다.

| 증빙 | 설명 |
| --- | --- |
| 요청 로그 | 요청 시각, API 경로, 대기열 토큰, 좌석 선점 시각 |
| 기기 로그 | device_id, device_fingerprint_hash, 앱/브라우저 버전 |
| 네트워크 로그 | IP, ASN, 프록시/VPN 의심 여부, 지역 변동 |
| 결제 로그 | PG 거래 식별값, 결제수단 fingerprint, 결제 실패 반복 |
| 행동 로그 | 마우스/터치 이벤트, 입력 속도, 화면 체류 시간 |
| 티켓 로그 | 구매 좌석, 재판매 등록, QR 열람, 입장 검증 이력 |

자동 탐지 결과만으로 최종 제재를 확정하지 않는다. 정상 사용자 피해를 줄이기 위해 고객센터 소명 절차를 제공하고, 운영자가 로그와 정황을 검토한 뒤 최종 조치를 결정한다. 단, 공연 당일 입장 질서와 다른 사용자 피해가 우려되는 고위험 계정은 입장 QR 활성화를 임시 보류할 수 있다.

## 12. 양도 결제, 정산 및 충전금 관리

초기 MVP와 베타에서는 양도/양수 결제를 충전금 전용으로 제한하지 않고, 카드, 간편결제, 계좌이체 등 일반 PG 결제를 기본으로 한다. 충전금은 빠른 매칭, 빠른 정산, 팬 월렛 같은 고도화 기능으로 추후 검토한다.

### 12.1 양도 결제수단 비교

양도 결제 방식은 크게 충전금 방식과 카드/일반결제 방식으로 나눌 수 있다.

| 방식 | 장점 | 문제점 |
| --- | --- | --- |
| 충전금 방식 | 플랫폼 안에서 돈 흐름을 통제하기 좋다. 양수자가 미리 충전해두면 매칭과 양도 완료가 빠르고, 양도자 정산을 플랫폼 원장으로 관리하기 쉽다. 충전 출처, 사용 내역, 정산 이력을 연결해 볼 수 있다. | 선불전자지급수단 이슈가 생길 수 있다. 충전금 별도관리, 환급 약관, 미사용 잔액 관리가 필요하다. 사용자가 왜 미리 충전해야 하는지 불편하게 느낄 수 있고, 환불, 현금 전환, 정산 민원이 생긴다. 초기 1인 기업에게 운영, 법무, 정산 부담이 크다. |
| 카드/일반결제 방식 | 사용자가 익숙하다. 충전, 환급, 잔액 관리가 필요 없고, PG 결제 승인, 취소, 환불 흐름으로 처리할 수 있다. 초기 MVP와 베타 출시에는 훨씬 단순하며 법무와 회계 리스크가 상대적으로 낮다. | 매칭 순간마다 결제 승인이 필요하다. 결제 실패 시 매칭 취소와 재매칭 로직이 필요하고, 양도자 정산은 별도 정산 시스템으로 처리해야 한다. 인기 티켓 매칭에서 결제 실패가 자주 나면 사용자 경험이 흔들릴 수 있다. |

따라서 현실적인 추천 방향은 다음과 같다.

```text
초기/베타:
양수자 = 카드, 간편결제, 계좌이체 등 일반 PG 결제
양도자 = 판매 확정 후 정산 계좌로 지급
플랫폼 = 양도 완료 시 공식 양도 수수료 차감

고도화 후:
충전금/보유금 옵션 추가 검토
빠른 매칭, 빠른 정산, 팬 월렛 기능으로 확장
```

### 12.2 양도 수수료 및 정산 정책

추천 방향은 양도 완료 시점에 공식 양도 수수료를 투명하게 받는 방식이다. 충전금 현금 전환 수수료를 핵심 수익원으로 삼지 않고, 초기에는 일반 PG 결제와 정산 계좌 지급 구조로 단순하게 운영한다.

권장 수수료 기준은 다음과 같다.

| 항목 | 권장안 |
| --- | --- |
| 양도 가격 | 원 티켓 금액의 0~10% 범위 안에서만 조정 |
| 공식 양도 수수료 | 거래 금액의 3~5% 또는 건당 1,000~2,000원 |
| 양도자 정산 | 판매 확정 후 정산 계좌로 지급 |
| 빠른 정산 수수료 | 정식 출시 후 선택 기능으로만 500~1,000원 또는 1% 이내 검토 |
| 충전금 | 초기 필수 기능에서 제외하고 추후 선택형 기능으로 검토 |

양수자가 매칭을 확정하기 전에는 티켓 양도 금액, 공식 양도 수수료, 최종 결제 금액을 모두 표시해야 한다. 결제 실패가 발생하면 해당 매칭은 즉시 완료 처리하지 않고, 결제 재시도 또는 재매칭으로 전환한다.

### 12.3 충전금 옵션 도입 시 관리 기준

충전금은 초기 필수 기능에서 제외하지만, 추후 빠른 매칭이나 팬 월렛 기능으로 도입할 수 있다. 이 경우 선불전자지급수단 이슈가 생길 수 있으므로 환급 조건과 수수료를 명확히 약관에 적어야 한다. 전자금융거래법은 선불전자지급수단 환급 약정을 약관에 기재하도록 하고 있고, 금융위원회는 선불충전금 보호와 별도관리 의무를 강화해왔다. 공정거래위원회도 신유형 상품권 표준약관 개정과 불공정 약관 시정 사례에서 환불 제한, 불명확한 환불수수료, 과도한 환급 제한을 문제 삼은 바 있다.

충전금 옵션 도입 시 필요한 관리 항목은 다음과 같다.

| 항목 | 설명 |
| --- | --- |
| wallet_charges | 충전 시점, 사용자, 금액, PG 거래 ID, 결제수단 fingerprint 기록 |
| wallet_ledger | 충전, 사용, 환불, 정산을 모두 원장형으로 기록 |
| payment_method_fingerprint | 카드번호 원문이 아니라 PG가 제공하는 식별값 또는 내부 해시 사용 |
| 환불 원칙 | 원 결제수단 환불을 기본으로 하여 자금 세탁성 이동 방지 |
| 충전 한도 | 신규 계정, 위험 계정, 동일 결제수단별 일/월 한도 적용 |
| 계정 간 이동 제한 | 보유금 선물, 계정 간 송금 기능은 초기에는 제공하지 않음 |

특히 공정위의 신유형 상품권 이용약관 시정 사례에서는 충전 후 일정 기간 안에는 수수료 없이 청약철회가 가능해야 하는데도 환불수수료를 부과하는 조항, 환불수수료를 “내부 규정에 따름” 또는 “일정 금액”처럼 불명확하게 적는 조항, 미사용 상품권 환불을 현금이 아닌 포인트나 캐시로만 제한하는 조항이 문제로 지적됐다. 공연 유료 멤버십 약관 시정 사례에서도 일정 기간 경과 또는 일부 혜택 사용만으로 전액 환불을 막거나 과다한 공제금을 적용하는 방식은 소비자에게 과도한 부담을 줄 수 있다는 취지로 개선됐다.

## 14. 수익 모델

Ticketground의 수익은 단순 예매 수수료만으로 구성하면 PG 수수료와 운영비를 감당하기 어렵다. 따라서 복수 수익원을 설계해야 한다.

| 수익원 | 설명 |
| --- | --- |
| 예매 수수료 | 티켓 구매 건당 고정 수수료 또는 티켓 가격 대비 일정 비율 |
| B2B 판매 대행 수수료 | 주최사 또는 판매 위탁사로부터 판매 대행 수수료 수취 |
| 공식 양도 수수료 | 일반 PG 결제 기반 양도 완료 시점에 거래 금액의 3~5% 또는 건당 1,000~2,000원 수준으로 투명하게 부과 |
| 공식 재판매 수수료 | 재판매 매칭 시 구매자 또는 판매자에게 합리적 수수료 부과 |
| 광고/프로모션 | 오픈 예정, 할인 예정, 단독 판매 노출 상품과 관심 공연 기반 예매 오픈 알림 |
| 프리미엄 운영 도구 | 대형 공연 주최사 대상 고급 통계, 좌석 관리, 리스크 분석 제공 |
| 데이터 리포트 | 개인정보를 제거한 판매 추세, 좌석 선호도, 카테고리별 수요 리포트 |

중요한 점은 취소 수수료나 충전금 현금 전환 수수료를 주 수익원으로 설계하지 않는 것이다. 취소 수수료와 환급 수수료는 소비자 분쟁과 규제 리스크가 크므로, 플랫폼의 핵심 수익은 판매 대행, 예매, 공식 양도, 공식 재판매, 운영 솔루션에서 만들어야 한다. 충전금은 초기 수익모델이 아니라 향후 빠른 매칭, 빠른 정산, 팬 월렛 옵션으로 검토한다.

## 15. 외부 연동 계획

### 15.1 PG 연동

실제 서비스 출시 전에는 PG 연동이 필요하다.

필수 기능은 다음과 같다.

| 기능 | 설명 |
| --- | --- |
| 결제 승인 | 카드, 계좌이체, 간편결제, 휴대폰 결제 |
| 결제 취소 | 전체 취소, 부분 취소, 취소 수수료 정책 반영 |
| 환불 | 원 결제수단 환불, 무통장 환불 계좌 검증 |
| 정산 | 주최사 정산, 재판매 판매자 정산 |
| 결제수단 식별 | 카드번호 원문 저장 없이 fingerprint 또는 billing key 활용 |

### 15.2 카카오톡, 문자 및 상담 연동

카카오톡과 문자는 관심 공연 예매 오픈 알림, 고객 문의 알림, 입장 QR 준비 및 활성화 안내에 사용한다. 단, 실제 QR 이미지는 전달하지 않고 예매 상세 화면, 전용앱 딥링크, 앱 설치 안내 링크로만 연결한다.

외부 채널별 역할은 다음과 같이 구분한다.

| 채널 | 역할 |
| --- | --- |
| 관심 공연 알림 | 예매 오픈 D-3, 오픈 당일, 상태 변경 알림을 사용자가 동의한 채널로 발송 |
| 입장 QR 알림 | 공연 1일 전 준비 안내, 공연 2~3시간 전 앱 딥링크 알림 발송 |
| 플랫폼 내부 문의함 | 예매번호, 결제, 환불, 양도, 입장 QR처럼 로그인과 거래 내역 확인이 필요한 문의 처리 |
| 카카오톡 채널 | 가입 전 문의, 단순 사용 문의, 공연 안내, 긴급 상담 진입 창구 |
| 카카오톡/문자 운영자 알림 | 문의 접수, 답변 필요, 공연 당일 긴급 문의를 운영자에게 알림 |
| 카카오 상담톡/CS SaaS | 상담량 증가 후 상담 배정, 태그, 자동응답, 통계 관리 |
| 자체 실시간 채팅 앱 | 초기에는 비추천. 정식 앱에는 고객센터 진입과 문의 내역 확인만 제공 |

1인 운영 단계에서는 사용자가 언제든 문의를 남길 수 있게 하되, 모든 문의가 즉시 실시간 응대되는 것처럼 안내하지 않는다. 화면 문구는 “운영시간 내 빠른 답변”, “공연 당일 입장 문의 우선 응대”, “답변 완료 시 알림 발송”처럼 운영 가능한 수준으로 설계한다. 상세 고객센터 정책은 5.8과 7.7, 입장 QR 알림 기준은 10.6을 따른다.

## 18. 암표 우회 시나리오 및 대응 정책

Ticketground의 암표 대응은 하나의 기능으로 끝나는 것이 아니라, 예매, 양도, 재판매, QR, 계정, 결제, 고객센터 예외 처리를 함께 묶어서 운영해야 한다. 사용자는 불편하지 않게 두되, 암표상이 플랫폼 밖에서 웃돈을 받고 플랫폼 안 기능을 우회 수단으로 쓰는 경우를 줄이는 것이 목표다.

주요 우회 시나리오와 대응 정책은 다음과 같다.

| 우회 시나리오 | 위험 | 대응 정책 |
| --- | --- | --- |
| 계정 통째 양도 | 암표상이 티켓이 있는 계정 자체를 넘긴 뒤 공연 후 비밀번호를 회수할 수 있음 | 공연 1일 전 이후 비밀번호 변경, 연락처 변경, 새 기기 로그인, QR 열람 기기 변경이 겹치면 OTP, 고객센터 확인, QR 임시 보류 적용 |
| 다매 구매 티켓 쪼개 판매 | 1인 구매 제한 안에서 여러 장을 산 뒤 각 티켓 QR을 다른 사람에게 따로 넘길 수 있음 | 대표 구매자 중심 동반 입장을 기본으로 하고, 티켓별 개별 이전은 공식 양도 기능으로만 허용 |
| 조건부 랜덤 매칭 조건 과도 축소 | 구매자가 좌석 등급, 구역, 가격, 연석 조건을 너무 좁혀 외부에서 약속한 좌석을 사실상 지정 거래할 수 있음 | 매칭 가능한 티켓이 최소 풀 크기 미만이면 매칭 보류 또는 조건 완화 유도 |
| 외부 프리미엄 정산 | 플랫폼 안에서는 0~10% 가격 정책을 지켜도 외부 계좌로 웃돈을 따로 받을 수 있음 | 신고 기능, 반복 양도자 감시, 동일 공연 반복 거래 탐지, 분쟁 발생 시 감사 원장과 결제 로그 확인 |
| 현장 고객센터 예외 악용 | 앱 설치 불가, 배터리 방전, 해외 사용자, 앱 장애를 핑계로 우회 입장을 시도할 수 있음 | 예외 입장은 본인확인, 예매정보, 결제수단 일부 확인, 운영자 승인, 1회용 처리 로그를 필수로 기록 |
| QR 화면 외부 촬영 | 캡처 차단을 우회해 다른 휴대폰으로 QR 화면을 촬영할 수 있음 | 20초 만료, 1회 사용, 동적 워터마크, QR 발급 로그, 위험 계정 보류로 추적과 무력화 병행 |
| 생체인증 우회 | 사용자가 기기 생체정보를 변경하거나 기기 잠금 방식을 공유해 실제 예매자가 아닌 사람이 QR을 열 수 있음 | 생체정보 원본은 수집하지 않고 OS 인증 결과와 앱 내부 기기 키 서명을 사용, 키 무효화 시 OTP 또는 고객센터 확인 후 신뢰 기기 재등록 |
| 충전금 우회 구매 | 결제 시점 카드 fingerprint 대신 미리 충전한 보유금으로 여러 계정이 구매할 수 있음 | 충전 시점의 결제수단 fingerprint, wallet ledger, 신규 계정 충전 한도, 충전 직후 인기공연 구매 패턴 탐지 |
| 고객센터 사회공학 | 휴대폰 고장, 인증 불가, 앱 오류를 이유로 상담원을 속여 QR 또는 입장 권한을 풀게 만들 수 있음 | 상담원 임의 해제 금지, 예외 처리 관리자 승인, 상담 로그 보관, 반복 예외 요청 계정 위험 점수 반영 |

조건부 랜덤 매칭에는 최소 풀 크기를 둔다. 예를 들어 구매자가 선택한 조건에 맞는 재판매 티켓이 3~5개 미만이면 즉시 매칭하지 않고, 조건을 완화하거나 대기 상태를 유지하도록 안내한다. 이 기준은 공연 규모와 좌석 등급별 재판매량에 따라 관리자가 조정할 수 있다.

```text
조건에 맞는 티켓 수 >= 최소 풀 크기:
조건부 랜덤 매칭 진행

조건에 맞는 티켓 수 < 최소 풀 크기:
매칭 보류
조건 완화 안내
또는 더 많은 재판매 티켓이 등록될 때까지 대기
```

계정 양도 대응은 공연 임박 시점에 더 강하게 적용한다. 평소에는 새 기기 로그인을 추가 인증으로 통과시킬 수 있지만, 공연 1일 전 이후에는 비밀번호 변경, 연락처 변경, 새 기기 로그인, QR 열람 기기 변경이 동시에 발생하면 계정 양도 가능성이 높다고 보고 QR 활성화를 보류할 수 있다.

1인 다매 구매는 동반 입장 편의를 위해 허용하되, 티켓별 QR을 여러 사용자에게 자유롭게 분리 배포하는 방식은 허용하지 않는다. 대표 구매자가 현장에서 동반자와 함께 입장하는 구조를 기본으로 하고, 동반자 중 일부가 따로 입장해야 한다면 공연 전 공식 양도로 해당 티켓의 소유권을 이전해야 한다. 동일 계정의 여러 티켓 QR이 짧은 시간 안에 서로 다른 기기, IP, 지역에서 열람되면 위험 점수를 높인다.

현장 예외 입장은 사용자를 보호하기 위해 필요하지만, 암표 우회 통로가 되지 않게 제한한다.

| 예외 처리 항목 | 필수 조건 |
| --- | --- |
| 앱 설치 불가 | 신분증 또는 본인확인, 예매정보, 결제수단 일부 확인 |
| 배터리 방전 | 예매자 본인확인, 예매번호, 휴대폰 번호 또는 로그인 계정 확인 |
| 앱 장애 | 운영자가 장애 모드로 전환한 경우에만 임시 웹 QR 또는 수기 확인 허용 |
| 해외 사용자 | 사전 앱 설치 안내를 발송하고, 실패 시 현장 본인확인 후 1회 처리 |

우회 대응의 원칙은 다음과 같다.

```text
정상 사용자는 추가 인증으로 통과시킨다.
고위험 사용자는 QR 활성화를 임시 보류한다.
확정 암표상은 거래 취소, 계정 차단, 결제수단 제한, 신고 검토 대상이 된다.
모든 제재는 자동 탐지 결과만으로 확정하지 않고 운영자 검토와 로그 증빙을 남긴다.
```

## 19. 기술적 한계와 대응

| 한계 | 대응 |

| 계정 양도 가능성 | 위험 기반 인증, 신뢰 기기, 공연 임박 정보 변경 감지 |

## 21. 결론

Ticketground는 단순히 티켓을 판매하는 플랫폼이 아니라, 예매, 공식 재판매, 입장 QR, 계정 위험 관리, 고객 문의, 운영 콘솔까지 연결하는 티켓 유통 통제 플랫폼을 목표로 한다.

현재 MVP는 사용자 예매 흐름과 관리자 운영 흐름의 핵심 골격을 이미 갖추고 있다. 다음 단계에서는 실제 결제, 회원 인증, 운영 DB, 카카오 상담 연동, 입장 QR 보안 정책 정합화, 현장 게이트 검증을 붙이면 베타 서비스 형태로 발전시킬 수 있다.

가장 중요한 방향은 다음과 같다.

```text
사용자는 편하게 예매한다.
티켓은 플랫폼 안에서만 안전하게 유통된다.
입장권은 공연 임박 시점에만 동적으로 열린다.
운영자는 모든 판매, 재판매, QR, 계정, 문의 이력을 확인한다.
```

이 구조를 유지하면 Ticketground는 예매 편의성과 암표 방지 기능을 동시에 내세울 수 있는 서비스로 성장할 수 있다.
